# 类和对象

## 一句话理解

类描述数据和行为，对象是类的实例。面试重点是对象布局、生命周期和资源复制，而不是类语法本身。

## 面试主线

- 对象里通常有什么，成员函数和静态成员在哪里。
- 构造、析构、拷贝构造、拷贝赋值分别在何时调用。
- 初始化列表为什么比构造函数体赋值更接近“初始化”。
- 裸指针成员为什么会引出深拷贝、浅拷贝和三法则。

## 对象模型

对象通常包含非静态数据成员、对齐填充；有虚函数时，常见 ABI 实现还会加入虚表指针 `vptr`。

| 位置 | 内容 |
| --- | --- |
| 对象内 | 非静态数据成员、对齐填充、可能存在的 `vptr` |
| 对象外 | 普通成员函数代码、静态成员变量、静态成员函数、虚函数表 |

- 普通成员函数不存进每个对象；调用时会隐式传入 `this` 指针。
- 静态成员变量属于类，不属于某一个对象；静态成员函数没有 `this`，不能访问非静态成员。
- 空类大小通常为 1 字节，用来保证不同对象有不同地址。
- `vptr`、虚表位置和对象大小属于编译器 ABI 细节，面试中应说“通常如此”，不能当作标准强制规则。

## 生命周期与默认成员函数

| 场景 | 调用 |
| --- | --- |
| 创建对象 | 构造函数 |
| 对象离开作用域或被 `delete` | 析构函数 |
| 用已有对象初始化新对象，如 `A b = a` | 拷贝构造 |
| 两个对象都已存在，再执行 `b = a` | 拷贝赋值 |

核心判断：**是否创建新对象**。创建了新对象就是构造或拷贝构造；已有对象被覆盖才是赋值。

```cpp
A a;
A b = a;  // 拷贝构造
A c;
c = a;    // 拷贝赋值
```

默认生成的拷贝操作会逐成员复制。对直接管理资源的类，这往往只复制指针地址，风险很高。

## 初始化列表

初始化列表在对象构造阶段直接初始化成员；构造函数体执行时，成员已经完成初始化，再写的是赋值。

```cpp
class User {
public:
    User(int id, const std::string& name)
        : id_(id), name_(name) {}
private:
    const int id_;
    std::string name_;
};
```

以下成员必须或应优先使用初始化列表：

- `const` 成员和引用成员必须初始化。
- 没有默认构造函数的成员对象必须在这里构造。
- 成员对象有合适构造函数时，可避免“默认构造后再赋值”。

成员实际初始化顺序只由其在类中的**声明顺序**决定，与初始化列表书写顺序无关。

## 资源复制：深拷贝与浅拷贝

浅拷贝复制成员值。若成员是裸指针，两个对象会指向同一资源，可能出现互相影响、悬空指针和双重释放。

深拷贝则重新申请资源并复制内容，让两个对象独立拥有资源。

```cpp
class Buffer {
public:
    Buffer(const Buffer& other)
        : data_(new int(*other.data_)) {}
    ~Buffer() { delete data_; }

private:
    int* data_;
};
```

如果类直接管理资源，通常要同时关注析构、拷贝构造和拷贝赋值，这就是三法则。现代 C++ 中应优先让成员类型自己管理资源，例如使用 `std::string`、容器或智能指针。

## 常见边界

- `class` 默认 `private`，`struct` 默认 `public`；这只是访问控制差异。
- `explicit` 构造函数禁止不期望的隐式转换。
- `this` 是非静态成员函数的隐式参数，不是每个对象额外存放的成员。
- 引用必须初始化且不能改绑；指针可以为空、可以改指向。
- 成员函数定义在类内会隐式 `inline`；是否真的内联由编译器决定。

## 我的薄弱点

- 对象内存布局曾误以为包含“成员函数指针”；应记住普通成员函数在对象外、通过 `this` 访问当前对象。
- 已能区分拷贝构造和拷贝赋值；后续表达要持续强调“是否创建新对象”。
- 深浅拷贝应说清具体风险：资源共享、双重释放和悬空指针。
- 初始化列表要抓住“直接初始化”，不要泛化成普通赋值语法。
