# C++设计模式

## 学习原则

模式是解决反复出现的设计问题的经验，不是代码模板。面试优先说明：**什么问题、何时使用、代价是什么**；简单场景不要为使用模式而增加层次。

## 工厂模式：集中对象创建

直接 `new` 会让业务代码依赖具体类，创建参数和类型选择散落各处。工厂将创建逻辑集中，调用方只依赖抽象接口。

| 类型 | 适用 |
|---|---|
| 简单工厂 | 一个入口按参数创建少量产品，如按配置选 MySQL 或 Redis 客户端 |
| 工厂方法 | 每种产品有对应工厂，新增产品时扩展新类 |
| 抽象工厂 | 一次创建一组彼此匹配的产品，如同一主题的整套 UI 组件 |

创建逻辑简单且类型固定时，直接构造通常更清晰。

## 单例模式：唯一实例

目标是一个类型在进程内只有一个实例，并提供统一访问点，适合配置、日志器等共享服务。

```cpp
Logger& instance() {
    static Logger object;
    return object;
}
```

C++11 保证函数内静态对象的**初始化**线程安全，可避免懒加载时多个线程重复创建实例；但不自动保证 `Logger` 的业务操作线程安全。单例会引入全局状态与隐藏依赖，应避免滥用。

## 策略模式：替换行为

将可变算法封装为统一策略接口，由上下文在运行时选择或替换。例如同一订单按满减、折扣、会员价等不同规则计算价格。

它与工厂的区别：工厂选择“**创建哪个对象**”，策略选择“对象创建后**采用什么行为**”。

## 观察者模式：事件通知

主题（发布者）维护观察者（订阅者）列表，事件发生时通知订阅者分别处理。适合订单状态变化后同步库存、积分、消息通知，或 GUI 事件与发布订阅。

重点风险：订阅者销毁前必须取消订阅，或使用弱引用等方式管理生命周期，避免悬空回调；事件链过长时也要关注顺序、失败与循环通知。

## 面试回答骨架

1. 工厂解耦创建与使用；简单工厂、工厂方法、抽象工厂分别覆盖单产品入口、可扩展产品、产品族。
2. 单例保证唯一实例。C++11 函数内静态对象初始化线程安全，但实例内部仍需自行同步。
3. 策略在运行时切换算法；工厂负责选择对象创建。
4. 观察者用于一对多事件通知，关键是订阅关系与生命周期管理。

## 本轮复盘

- 需强化：简单工厂、工厂方法、抽象工厂的层级区别。
- 需强化：观察者的主题/观察者职责和取消订阅的必要性。
