# C++异常

## 一句话理解

异常是程序执行期间发生的意外情况，C++通过 `try`/`catch`/`throw` 机制将正常代码与错误处理逻辑分离，提供一种结构化的错误处理方式。

## 核心原理

### 异常的基本概念

在C语言中，错误处理要么终止程序，要么返回错误码，使用不便。C++的异常机制提供了一种更优雅的处理方式。

**核心概念**：

| 概念 | 说明 |
|------|------|
| 异常抛出（Throw） | 使用 `throw` 关键字抛出一个异常对象 |
| 异常捕获（Catch） | 使用 `try-catch` 语句块捕获并处理异常 |
| 异常对象 | 用于表示特定异常情况的C++对象（内置类型、自定义类型或标准异常类） |
| 栈展开（Stack Unwinding） | 抛出异常后，沿函数调用链向上查找匹配的 catch 块 |

### 异常的抛出和匹配原则

**抛出原则**：
- 仅在出现无法恢复或处理的错误时抛出异常（如内存分配失败、打开文件失败等）
- 抛出的异常应与问题本身相关，使用具体的异常类型或其派生类
- 异常对象可以是内置类型、自定义类型或标准异常类的实例

**匹配原则**：
- 使用 `try` 块包装可能引发异常的代码
- `catch` 块按出现顺序匹配异常类型
- 应将更具体的异常类型放在更前面的 `catch` 块中

```cpp
try {
    // 可能引发异常的代码
} catch (const std::runtime_error& e) {
    // 处理运行时错误
} catch (const std::exception& e) {
    // 处理标准异常
} catch (...) {
    // 捕获任意类型的异常
}
```

### 函数调用链中异常栈展开

当异常被抛出时，程序从当前函数开始展开函数调用栈，寻找匹配的 `catch` 块：

1. 从当前函数开始查找匹配的 `catch` 块
2. 如果当前函数没有合适的 `catch` 块，异常传递给调用者
3. 重复直到找到匹配的 `catch` 块或到达 `main()` 函数
4. 如果到达 `main()` 仍未匹配，程序终止（建议最后加 `catch(...)` 兜底）
5. 在栈展开过程中，局部对象的**析构函数会被自动调用**，完成资源清理

```cpp
void funcB() {
    throw std::runtime_error("Exception in funcB");
}
void funcA() { funcB(); }
int main() {
    try { funcA(); }
    catch (const std::exception& e) {
        std::cout << e.what() << std::endl;
    }
    return 0;
}
```

### 异常的重新抛出

在 `catch` 块中使用 `throw;`（不带参数）将当前捕获的异常重新抛出，允许在多个层级的代码中进行适当的处理或传播。

```cpp
try {
    // ...
} catch (const std::exception& e) {
    // 记录日志
    throw;  // 重新抛出，交给上层处理
}
```

## 底层实现

### 异常安全级别

| 级别 | 保证 | 说明 |
|------|------|------|
| 基本异常安全 | 无资源泄漏 | 异常发生时资源被正确释放 |
| 强异常安全 | 状态不变 | 异常发生时对象回滚到之前的状态 |
| 不抛出异常 | 绝不抛出 | 函数在任何情况下都不会抛出异常 |

### 异常安全实践

1. **RAII（资源获取即初始化）**：在构造函数中获取资源，析构函数中释放资源。异常发生时，析构函数自动调用，确保资源释放
2. **构造函数中避免抛出异常**：否则可能导致对象不完整或未完全初始化
3. **析构函数中避免抛出异常**：否则可能导致资源泄漏；如果析构函数中必须处理，应捕获并处理，不应传播出去
4. **智能指针**：使用 `std::unique_ptr`、`std::shared_ptr` 管理动态内存，避免内存泄漏

### 异常规范（已弃用）

C++98支持在函数声明中指定可能抛出的异常类型：
```cpp
void Function() throw(ExceptionType1, ExceptionType2); // C++98风格
```

在 C++11 中已不再是强制性的语言特性。现代 C++ 推荐使用 RAII 和异常安全编程技术替代异常规范。

## 常见应用场景

- **资源分配失败**：如 `new` 失败抛出 `std::bad_alloc`
- **文件操作错误**：文件打开失败、读写错误
- **数据验证**：输入参数不合法、数据范围越界
- **类型转换失败**：`dynamic_cast` 转换失败
- **数学运算错误**：除零错误、溢出等

### 自定义异常体系

通过继承 `std::exception` 创建自定义异常类，提供更具语义的错误信息：

```cpp
class MyAppException : public std::exception {
public:
    MyAppException(const char* message) : message_(message) {}
    const char* what() const noexcept override { return message_; }
private:
    const char* message_;
};
```

### C++标准异常类

| 异常类 | 说明 |
|--------|------|
| `std::exception` | 所有标准异常类的基类，定义 `what()` 函数 |
| `std::bad_alloc` | 内存分配失败 |
| `std::bad_cast` | `dynamic_cast` 失败 |
| `std::logic_error` | 逻辑错误（程序逻辑问题） |
| `std::runtime_error` | 运行时错误（底层系统或环境问题） |
| `std::overflow_error` | 算术溢出 |
| `std::underflow_error` | 算术下溢 |
| `std::range_error` | 超出有效范围 |

## 容易踩坑的地方

1. **按值捕获异常**：应该使用引用捕获（`catch (const std::exception& e)`），避免对象切片和额外的拷贝开销
2. **析构函数中抛出异常**：析构函数中抛出异常且未捕获时，如果有另一个异常正在传播，会导致 `std::terminate` 被调用
3. **构造函数中抛出异常**：构造函数抛出异常时，对象析构函数不会被调用，已构造完成的成员会自动析构，但构造函数中已分配的资源需要自行处理
4. **catch 块顺序错误**：如果将基类异常的 `catch` 放在派生类异常的 `catch` 前面，派生类异常永远不会被匹配到
5. **过度使用异常**：不适合预期内的控制流，也不适合嵌入式系统或实时系统
6. **异常性能开销**：抛出异常时栈展开操作有较大性能开销，不应用于频繁调用的场景
7. **忘记添加 catch(...)**：主函数中没有兜底捕获时，未处理的异常会导致程序直接终止

## 面试高频问题

1. C++异常的抛出和捕获机制？—— throw 抛出，try-catch 捕获
2. 什么是栈展开（stack unwinding）？—— 异常沿调用链向上查找 catch 块的过程，局部对象析构函数自动调用
3. 异常安全有哪些级别？—— 基本异常安全、强异常安全、不抛出异常
4. RAII 如何帮助实现异常安全？—— 资源在构造函数中获取，析构函数中释放，异常时析构函数自动调用
5. 构造函数和析构函数中抛出异常有什么问题？—— 构造函数抛出对象不完整；析构函数抛出且处于异常传播中会导致 terminate
6. 自定义异常类如何实现？—— 继承 std::exception，重写 what() 函数
7. `catch(...)` 的作用？—— 捕获任意类型的异常，作为兜底处理
8. 为什么要用引用捕获异常？—— 避免对象切片和额外拷贝
