# C++异常

## 一句话理解

异常用于处理当前层难以恢复的、非预期的失败；抛出后通过栈展开自动析构局部 RAII 对象。高频且预期内的失败，通常用错误码、`optional` 或 `expected` 表达更清晰。

## 执行流程与栈展开

```cpp
try {
    work();
} catch (const std::exception& e) {
    log(e.what());
}
```

1. `throw` 创建并抛出异常对象。
2. 运行时沿调用栈向上寻找第一个匹配的 `catch`。
3. 每退出一层函数，已构造的局部对象按构造的反序析构，这个过程称为**栈展开**。
4. 没有匹配的 `catch` 时，调用 `std::terminate()`。

这正是 RAII 的价值：资源被局部对象管理时，无论正常返回还是异常离开作用域，析构函数都会释放锁、文件句柄和堆内存等资源。

## 捕获、匹配与重新抛出

优先使用 `const` 引用捕获：

```cpp
catch (const std::exception& e) { /* ... */ }
```

它避免额外复制，也避免按基类值捕获派生异常时发生对象切片。`catch` 按书写顺序匹配，因此具体派生异常必须写在基类异常之前，`catch (...)` 只作最后兜底。

```cpp
catch (const std::out_of_range& e) { /* 具体处理 */ }
catch (const std::exception& e)    { /* 兜底 */ }
```

在 `catch` 内：

- `throw;` 原样重新抛出当前异常，保留原始动态类型；记录日志或局部清理后继续向上传播时使用。
- `throw e;` 抛出一个新的、按 `e` 静态类型构造的异常；可能复制或切片，通常不选。

## `noexcept` 与性能

```cpp
void release_resource() noexcept;
```

`noexcept` 表示异常不能从函数边界传播出去。函数内部可以抛出并自行捕获；若异常逃出 `noexcept` 函数，直接调用 `std::terminate()`。

它也是接口信息：如 `std::vector` 扩容时，元素移动构造为 `noexcept`，容器才更可能选择移动而非拷贝，以维持异常安全。

现代实现通常让正常路径几乎没有异常处理的运行时开销，但会增加展开元数据和少量代码体积；真正昂贵的是异常发生后的对象构造、查找处理器和跨栈展开。因此不要用异常表达高频正常分支。

## 异常安全保证

| 级别 | 异常发生后 |
| --- | --- |
| 无保证 | 状态可能损坏，也可能泄漏资源 |
| 基本保证 | 不泄漏资源，对象仍合法可用，但状态可能改变 |
| 强保证 | 操作要么完全成功，要么状态回到操作前 |
| 不抛保证 | 异常不从函数边界传播，通常用 `noexcept` 表示 |

RAII、先完成可能失败的操作再提交状态（commit-or-rollback）、以及智能指针是实现异常安全的常见手段。

构造函数抛异常是合法的：对象本身不会被析构，但已构造完成的成员和基类子对象会自动析构。避免用裸资源让构造函数中途失败后无法清理。

析构函数应避免传播异常；尤其在栈展开期间再次抛出，会直接 `std::terminate()`。

## 异常还是错误返回

| 场景 | 更合适的方式 |
| --- | --- |
| 预期内、常发生、需要调用方分支处理 | 错误码、`optional`、`expected` |
| 仅表示“可能没有值”，不需要错误原因 | `optional<T>` |
| 需要同时传递值或具体错误 | `expected<T, E>` |
| 当前层无法合理恢复的失败 | 异常 |
| 实时/嵌入式/禁异常项目、跨 C 接口 | 错误码或结果类型 |

例如“未命中缓存”通常是正常分支；内存分配失败或构造过程中无法建立对象则更像异常。

## 自定义异常

业务异常一般继承 `std::exception` 并重写 `what()`：

```cpp
class ConfigError : public std::runtime_error {
public:
    using std::runtime_error::runtime_error;
};
```

常见标准异常：`std::bad_alloc`（内存分配失败）、`std::out_of_range`（越界）、`std::runtime_error`（运行时错误）。
