# 智能指针

## 一句话理解

智能指针用 RAII 管理资源生命周期：`unique_ptr` 表示独占所有权，`shared_ptr` 表示共享所有权，`weak_ptr` 表示只观察不拥有。

## 核心原理

### RAII

RAII 的核心是：对象构造时接管资源，析构时释放资源。即使中途抛异常，栈对象析构也会自动执行，因此智能指针比手动 `new/delete` 更安全。

### 所有权模型

| 智能指针 | 所有权语义 | 典型用途 |
|----------|------------|----------|
| `unique_ptr` | 独占资源，不能拷贝，只能移动 | 工厂返回值、类成员独占资源 |
| `shared_ptr` | 多个指针共享资源，最后一个释放资源 | 多处共同使用同一对象 |
| `weak_ptr` | 只观察资源，不增加强引用计数 | 打破循环引用、判断对象是否还活着 |

### unique_ptr

`unique_ptr` 独占资源，拷贝构造和拷贝赋值被禁用，只能通过移动语义转移所有权，移动后源指针为空。它通常和原始指针一样轻量，默认应优先选择它。

### shared_ptr

`shared_ptr` 共享资源所有权。拷贝时强引用计数加一，析构或 `reset()` 时强引用计数减一；强引用计数归零时销毁被管理对象。

标准库实现通常不是简单的 `T* + int*`，而是 **指针 + 控制块**：

```text
shared_ptr
  ├── ptr            -> 对外表现为指向哪个对象
  └── control block  -> 管理生命周期
        ├── strong_count
        ├── weak_count
        ├── deleter
        └── allocator
```

控制块的意义：
- 保存强引用计数和弱引用计数。
- 支持 `weak_ptr` 判断对象是否还活着。
- 保存自定义删除器和分配器。
- 支持 `make_shared` 将对象和控制块合并分配。
- 支持别名构造，让“对外指向的对象”和“实际管理的对象”不完全相同。

### 手写版和标准库版的区别

手写教学版常见结构：

```text
SharedPtr
  ├── ptr
  └── count
```

这能说明引用计数的核心思想，但不足以完整支持 `weak_ptr`、自定义删除器、分配器和别名构造。

标准库版更接近：

```text
SharedPtr
  ├── ptr
  └── control block
```

所以不是手写版错了，而是手写版只覆盖“共享计数”这个核心点；标准库要覆盖完整工程场景。

### weak_ptr 的作用

`weak_ptr` 是观察者：它知道对象是否还活着，但不会延长对象生命周期。它不能直接访问对象，必须先通过 `lock()` 临时提升为 `shared_ptr`。

```cpp
std::weak_ptr<T> w = sp;

if (auto p = w.lock()) {
    // 对象还活着，临时获得 shared_ptr 使用
}
```

最典型用途是打破循环引用：

```text
A --shared_ptr--> B
B --shared_ptr--> A   // 循环引用，强引用计数无法归零

A --shared_ptr--> B
B --weak_ptr----> A   // B 只观察 A，不拥有 A
```

### weak_ptr 的实现思路

要实现 `weak_ptr`，`shared_ptr` 必须改成控制块模型：

```text
strong_count == 0：销毁对象
没有强引用且没有 weak_ptr：销毁控制块
```

`weak_ptr` 构造时只增加 `weak_count`，析构时只减少 `weak_count`。`lock()` 的逻辑是：如果 `strong_count > 0`，说明对象还活着，就把强引用计数加一并返回一个临时 `shared_ptr`；否则返回空指针。

多线程下，`lock()` 不能简单写成“先判断再加一”，因为中间对象可能被别的线程释放。真实实现需要用 CAS 等原子操作保证“检查并增加强引用计数”是一个不可被打断的整体。

### 引用计数为什么需要原子操作

多个线程可能同时拷贝或销毁**不同的 `shared_ptr` 副本**。如果引用计数加减不是原子的，就可能计数丢失、提前释放对象或重复释放对象。

注意：引用计数线程安全，不等于对象本身线程安全。多个线程同时修改 `*ptr` 指向的对象，仍然需要锁或其他同步机制；同一个 `shared_ptr` 变量若被并发 `reset` 或赋值，也需要外部同步。

### make_shared

`make_shared` 通常一次性分配控制块和对象，减少分配次数，通常也更有利于降低分配开销和碎片。

但它不能直接指定自定义删除器；并且若有长期存在的 `weak_ptr`，对象销毁后控制块仍保留，合并分配的对象存储通常也会延迟归还。直接 `shared_ptr<T>(new T, deleter)` 是两次分配，却能自定义删除策略，且对象本体会在强引用归零时先释放。

默认优先 `make_shared`，遇到上述边界再直接构造。

### `get()`、`shared_from_this()` 与所有权边界

`get()` 返回的是不拥有资源的借用指针：只能临时传给明确“不保存、不释放”的接口。

- 不能对 `get()` 的结果 `delete`。
- 不能再用它构造独立的 `shared_ptr`，否则会有两个控制块重复释放同一对象。
- `unique_ptr::release()` 才是放弃所有权的接口，调用后必须立即明确新的接管者。

成员函数中不能写 `shared_ptr<T>(this)`：对象可能在栈上，或已被另一个 `shared_ptr` 管理，都会导致错误释放或双控制块。需要获得自身的共享所有权时，继承 `enable_shared_from_this<T>`，并保证对象一开始就由 `shared_ptr` 管理：

```cpp
class T : public std::enable_shared_from_this<T> {
public:
    std::shared_ptr<T> self() { return shared_from_this(); }
};
```

### 函数参数如何表达所有权

| 参数形式 | 表达的语义 |
| --- | --- |
| `unique_ptr<T>` 按值 | 函数接管唯一所有权，调用方用 `std::move` 传入 |
| `shared_ptr<T>` 按值 | 函数成为共享所有者，可能跨调用保存对象 |
| `weak_ptr<T>` 按值 | 函数保存观察关系，使用时 `lock()` |
| `T*` / `T&` | 只在调用期间使用，不接管也不保存 |

只临时使用对象时，优先传 `T&` 或可空的 `T*`，不要为了“安全”随手传 `shared_ptr`。`unique_ptr` 能转换为 `shared_ptr`；反向通常不安全，因为无法保证没有其他共享所有者，`use_count()` 也只是瞬时快照。

## 容易踩坑的地方

1. 循环引用：两个对象互相持有 `shared_ptr` 会导致强引用计数无法归零，应将非拥有关系改成 `weak_ptr`。
2. 裸指针混用：同一资源不要交给多个独立智能指针管理。
3. 线程安全误解：`shared_ptr` 的引用计数线程安全，但被管理对象不自动线程安全。
4. `weak_ptr` 不能直接解引用，必须先 `lock()` 判断对象是否还活着。
5. 自定义删除器：`unique_ptr` 的删除器影响类型，`shared_ptr` 的删除器保存在控制块中。
6. `auto_ptr`：拷贝时会偷偷转移所有权并把源置空，语义反直觉且不适配容器；已被 `unique_ptr` 取代。
7. `get()` 是借用，不是所有权转移；不要 `delete` 它，也不要据此新建独立 `shared_ptr`。
8. `shared_ptr(this)` 会创建第二个控制块；需要自身共享所有权时使用 `enable_shared_from_this`。
