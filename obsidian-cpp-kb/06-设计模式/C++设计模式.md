# C++设计模式

## 一句话理解
设计模式是在软件设计中反复出现的问题的通用可复用解决方案，通过七大设计原则的指导，提供构建可维护、可扩展、松耦合代码的最佳实践。

## 核心原理

### 设计模式七大原则

| 原则 | 核心思想 |
|------|----------|
| **单一职责原则（SRP）** | 一个类只负责一个职责，只有一个引起变化的原因 |
| **开放封闭原则（OCP）** | 对扩展开放，对修改关闭 |
| **里氏替换原则（LSP）** | 子类必须能够替换父类而不影响程序正确性 |
| **依赖倒置原则（DIP）** | 依赖抽象而非具体实现，高层模块不依赖底层模块 |
| **接口隔离原则（ISP）** | 客户端不应依赖它不需要的接口 |
| **合成/聚合复用原则（CARP）** | 多用组合/聚合，少用继承 |
| **最少知道原则（LKP/迪米特法则）** | 一个对象应尽量减少对其他对象的了解 |

### 单例模式（Singleton）

确保一个类只有一个实例，并提供全局访问点。

**饿汉模式**：程序启动时即创建实例，线程安全，但可能造成资源浪费。

```cpp
class Singleton {
private:
    Singleton() {}
    static Singleton instance;
public:
    static Singleton& getInstance() { return instance; }
};
```

**懒汉模式**：首次使用时才创建实例，C++11后局部static变量初始化是线程安全的。

```cpp
class LazySingleton {
private:
    LazySingleton() {}
public:
    static LazySingleton& getInstance() {
        static LazySingleton instance;
        return instance;
    }
};
```

### 工厂模式（Factory）

封装对象的创建过程，实现创建和使用分离。

| 模式 | 特点 | 适用场景 |
|------|------|----------|
| **简单工厂** | 单一工厂类根据参数创建不同产品 | 产品种类较少，创建逻辑简单 |
| **工厂方法** | 将对象的创建推迟到子类，每个产品对应一个工厂 | 产品等级结构单一 |
| **抽象工厂** | 创建一系列相关或依赖的产品族 | 产品族需要保持一致性 |

### 建造者模式（Builder）
将一个复杂对象的构建过程与其表示分离，同样的构建过程可以创建不同的表示。

组成部分：产品类、抽象建造者、具体建造者、指挥者类。

### 代理模式（Proxy）
为其他对象提供代理以控制对该对象的访问，可在访问前后添加额外功能（权限控制、缓存、日志等）。

组成部分：抽象主题、真实主题、代理类。

## 底层实现

### 单例模式的线程安全
- 饿汉模式：静态变量在程序启动时初始化，本身线程安全
- 懒汉模式（C++11）：局部static变量的初始化由编译器保证线程安全

### 工厂模式的解耦原理
通过引入抽象接口和工厂类，客户端仅依赖抽象接口而非具体类，实现编译时解耦。新增产品时只需新增对应类和工厂，无需修改已有代码（满足OCP）。

## 常见应用场景

1. **单例模式**：日志系统、配置管理、数据库连接池
2. **工厂模式**：对象创建过程复杂、需要运行时决定创建哪种对象
3. **建造者模式**：复杂对象的构建（如HTML文档生成器）
4. **代理模式**：延迟加载、访问控制、日志记录、远程代理

## 容易踩坑的地方

1. **单例模式滥用**：引入全局状态，隐藏模块间依赖关系，不利于测试
2. **工厂模式过于复杂**：简单场景过度使用工厂模式增加不必要的复杂性
3. **违反里氏替换原则**：子类重写父类方法时改变原有行为（如企鹅继承鸟类的fly方法）
4. **接口隔离不当**：大而全的接口迫使实现类实现不需要的方法
5. **继承滥用**：优先考虑组合而不是继承
