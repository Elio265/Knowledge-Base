# SangBridge AI统一接入与管理控制台

> 本页是项目主档案。初始范围来自项目介绍，架构细节、需求结论和 TD 根因需在源码、需求文档或提交记录中逐项核验后再标记为已确认。

## 项目定位

SangBridge 面向 EDS 5.3.2，采用前后端一体化交付，为 AI 与基础设施服务提供统一的控制台和网关入口：

```text
用户
  -> SangBridge Web 控制台 / FastAPI 网关
  -> Uniview、VCS 等上游服务
```

它统一承接页面交互、身份认证、Token 管理、请求转发、安全控制和操作审计，避免用户分别面对多个上游服务的认证、入口和权限模型。

## 初始范围（待源码/需求核验）

| 模块 | 职责 |
|---|---|
| `web/` | Vue 3 控制台：登录、Uniview、VCS 等管理页面 |
| `server/` | FastAPI 网关：代理、鉴权、Token 持久化、审计、限流、WAF 适配和上游对接 |
| `docs/` | 架构、模块、版本需求、部署和运维文档 |
| `deploy/`、`.ci/` | 服务化/RPM 部署、代码检查、单元测试和契约测试 |

当前负责范围：TD 修改与需求处理。项目整理的重点不只记录功能，还要保留每个变更的影响面、设计取舍、验证证据和遗留风险。

## 架构问题树

后续按以下顺序回查，每个结论必须能对应源码、需求、接口契约或测试证据：

1. **入口与鉴权**：登录方式、Token 签发/刷新/持久化、前后端权限边界。
2. **网关转发**：路由如何选择上游，认证头和上下文如何透传，错误如何归一化。
3. **安全控制**：限流、WAF、审计、敏感信息脱敏、上游异常与重试边界。
4. **核心业务**：VCS 的仓库/分支/标签/提交/合并请求，以及 Uniview 的具体管理能力。
5. **交付运维**：配置来源、部署拓扑、健康检查、日志指标、升级/回滚与 CI 契约测试。

## TD 与需求的沉淀格式

每个 TD 或需求只在本页追加一条记录，先写事实再写结论：

| 字段 | 需要回答的问题 |
|---|---|
| 背景与目标 | 谁遇到什么问题，完成后用户或系统得到什么能力？ |
| 影响面 | 涉及 Web、网关、上游协议、配置、部署、权限或审计中的哪些部分？ |
| 根因/取舍 | 为什么当前行为出现，为什么选这个方案而不是替代方案？ |
| 实现与证据 | 关键模块、接口、提交、测试或日志证据是什么？ |
| 验证 | 正向、异常、权限、安全、兼容与回归如何验证？ |
| 风险与后续 | 哪些边界尚未覆盖，是否需要监控、文档或后续 TD？ |

> [!important] 事实边界
> “代码当前如此实现”“已上线验证”“建议改成这样”必须分开写。未经验证的内网查询结果只能作为排查线索，不能直接当作项目事实或面试承诺。

## 后续复习入口

- 1 分钟介绍：背景、统一入口价值、本人负责的 TD/需求、最有代表性的改动与结果。
- 高频追问：鉴权边界、Token 安全、网关转发失败、审计与隐私、限流/WAF 误伤、上游兼容、发布回滚。
- 薄弱点：每轮问答后记录不熟悉的模块和待核验路径，优先进入下一轮复测。
