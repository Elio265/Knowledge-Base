# NFS Kerberos 域接入与用户映射管控

> 项目定位：为 NFS `sec=krb5` 认证场景建设 Kerberos 相关管控面能力。重点是配置编排、跨节点下发、数据面热加载与失败补偿；**不是**从零实现 Kerberos 协议。

## 一句话介绍

从 0 到 1 建设 Kerberos 域接入与域用户—Unix 用户映射的管控面：管理员通过界面配置域控和 SPN 策略，管控面生成认证配置并持久化，通过 HTTP 下发至节点，再以 SIGHUP 通知数据面重载；任一阶段失败时执行对应的补偿与回滚。

## 已确认的事实

### 配置与管理能力

- 管理界面支持配置：域控 IP、端口、域名、管理员账号、管理员密码，以及是否覆盖已存在的 SPN。
- 支持域用户到 Unix 用户的映射；映射由管理员显式创建和维护。
- 管控面维护 MongoDB 中的配置数据，同时维护配置文件。
- 加域会生成或更新 `krb5.conf`，并创建 keytab；退域会删除 keytab。
- 目标使用场景与 NFS 相关，配置中出现 `sec=krb5`。

### 配置变更链路

```text
管理界面/API
  → 配置校验与任务编排
  → MongoDB 写入
  → 本地配置文件写入（含 krb5.conf / keytab 生命周期）
  → HTTP 配置下发至其他节点
  → SIGHUP（kill -1）通知数据面重新加载
```

- 管控面由本人从 0 开始开发，并负责与数据面对接。
- 使用 Python 实现；异步任务框架尚未确认。
- 变更过程按阶段执行：数据库写入、配置写入、节点分发、数据面通知。任一步失败，回滚该步骤及已完成的相关步骤。
- 这里应称为“分步补偿 / 回滚”，不要在不了解实现前称为“分布式事务”或“强一致事务”。

## 面试高频追问与准备

### 1. 为什么这个项目需要 Kerberos？与 NFS 有什么关系？

**回答边界：** 已确认目标场景使用 NFS `sec=krb5`。先解释 Kerberos 用于基于票据的身份认证，NFS 使用 `sec=krb5` 时需要相关认证配置和凭据；不要在未确认前扩展到具体 GSS、RPC 流程。

**要补的事实：** `sec=krb5` 的确切配置位置与参与进程。

### 2. 为什么要让管理员维护域用户—Unix 用户映射？

**回答方向：** NFS/Linux 权限校验最终依赖本地 Unix 身份和 UID/GID 语义；域身份需要有明确、可管理的本地映射关系。说明本项目选择“管理员显式维护”的原因及冲突处理规则。

**要补的事实：** 映射格式、落库位置、生效位置、冲突和删除处理。

### 3. 加域、退域具体做了什么？

**已确认回答：** 加域会配置 `krb5.conf` 并创建 keytab；退域会清理 keytab。前端还支持 SPN 是否覆盖的策略。

**要补的事实：** keytab 的创建者、权限、原子写入方式、SPN 操作的精确定义。

### 4. 为什么选 HTTP 做配置下发？如何保证可靠性？

**回答框架：** HTTP 便于管控面主动编排、节点接口清晰、可返回状态。但 HTTP 成功只能说明请求层面成功，仍需说明超时、重试、幂等、节点状态记录与部分失败策略。

**要补的事实：** 实际的超时、重试、鉴权和部分失败处理。没有依据不要说“最终一致”或“至少一次投递”。

### 5. 为什么用 SIGHUP，而不是直接重启数据面？

**回答方向：** SIGHUP 常用于通知长期运行的 Linux 服务重新读取配置，可避免中断服务。关键不是信号本身，而是数据面是否支持安全重载、如何校验重载结果、失败如何恢复。

**追问风险：** `kill -1` 成功不等于配置已生效；必须确认数据面是否上报成功状态或写入日志。

### 6. 回滚是怎么设计的？

**已确认回答：** 流程包括 MongoDB 写入、配置文件写入、节点 HTTP 分发、SIGHUP 通知；失败时按步骤补偿和回滚。

**建议表达：** 这是 Saga 风格的分步补偿，不是数据库 ACID 事务。描述一个具体失败案例：例如节点分发失败时，哪些已完成动作被撤销，如何保证重试不会把状态弄乱。

**要补的事实：** 回滚顺序、旧配置保存位置、每一步的幂等方式、部分节点失败策略。

### 7. 管理员密码和 keytab 如何保护？

**回答方向：** 这是安全敏感问题。只有确认后才回答具体加密、脱敏、权限和审计方案；当前至少应主动识别风险：密码不能明文写日志，keytab 应限制所有者和文件权限。

### 8. SPN 为什么可能冲突？“覆盖已有 SPN”意味着什么？

**回答方向：** SPN 是服务身份标识，重复或归属错误会导致客户端找到错误主体或认证失败。覆盖策略是高风险操作，应先校验冲突、明确归属并记录审计。

**要补的事实：** 实际 SPN 格式、冲突检测与覆盖操作的实现位置。

### 9. 你使用 AI 生成过代码，如何保证质量？

**建议如实回答：** AI 用于加速样板代码和实现探索；我负责把需求拆成状态和失败场景，决定配置链路与回滚边界，审阅生成代码，并通过接口联调、异常场景和交付验收确认结果。对安全、幂等和数据面协议相关代码不能只依赖生成结果，必须结合实际环境验证。

### 10. 你最大的技术难点是什么？

**建议答案：** 难点不是一个单点 API，而是跨数据库、文件系统、节点下发和数据面重载的状态管理。任一步失败都可能留下部分配置，因此需要明确步骤、成功判定、补偿动作和可观测性。
