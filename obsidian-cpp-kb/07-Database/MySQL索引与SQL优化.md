# MySQL 索引与 SQL 优化

## 一句话理解

MySQL 索引的核心目标是减少磁盘 IO。面试里重点不是背概念，而是能解释 B+ 树、聚簇索引、回表、覆盖索引、最左前缀、索引失效和 `EXPLAIN`。

## B+ 树为什么适合索引

InnoDB 常用 B+ 树做索引，核心原因是它适合磁盘页和范围查询。

| 对比 | 特点 |
|------|------|
| B+ 树 | 多叉、树高低、叶子节点有序链表，适合范围查询和排序 |
| 哈希表 | 等值查询快，但不适合范围查询、排序、最左前缀 |
| B 树 | 非叶子节点也存数据，单页能放的 key 较少，范围扫描不如 B+ 树方便 |

B+ 树优势：

- 多叉树，树高低，访问磁盘页次数少。
- 叶子节点按顺序连接，适合范围查询。
- 索引本身有序，可以支持部分 `order by`、分页、范围扫描。
- 非叶子节点主要存 key 和指针，一个页能容纳更多 key。

面试表达：

> MySQL InnoDB 常用 B+ 树做索引，主要是为了减少磁盘 IO。B+ 树是多叉平衡树，树高低，查找时访问的磁盘页少；叶子节点按顺序通过链表连接，天然支持范围查询和排序。哈希索引虽然等值查询快，但不适合范围和排序；B 树非叶子节点也存数据，单页能放的 key 更少，范围扫描也不如 B+ 树方便。

## 聚簇索引和二级索引

InnoDB 的主键索引是聚簇索引：

```text
主键 B+ 树叶子节点：
主键 id -> 整行数据
```

普通索引是二级索引：

```text
普通索引叶子节点：
索引列值 -> 主键 id
```

例子：

```sql
select * from user where id = 10;
```

按主键查，找到主键索引叶子节点就能拿到整行数据。

```sql
select * from user where name = 'Tom';
```

如果 `name` 是普通索引，流程通常是：

```text
先查 name 二级索引
拿到主键 id
再用 id 查主键索引拿整行数据
```

面试表达：

> InnoDB 的主键索引是聚簇索引，叶子节点存放整行数据，所以按主键查询可以直接拿到数据。普通二级索引的叶子节点存的是主键值，不存整行数据。如果通过二级索引查询的字段不够用，就需要先查二级索引拿到主键，再回到主键索引查整行，这个过程叫回表。

## 回表和覆盖索引

回表：

```text
二级索引 -> 主键 id -> 主键索引 -> 整行数据
```

覆盖索引：

```text
查询需要的字段都在二级索引里
不需要再回主键索引查整行
```

例子：

```sql
index idx_name_age(name, age)

select age from user where name = 'Tom';
```

这个查询只需要 `name` 和 `age`，都在 `idx_name_age` 里，可以不回表。

减少回表：

- 少用 `select *`，只查需要字段。
- 建合适的联合索引，让查询字段被索引覆盖。
- 让过滤条件和返回字段尽量走同一个索引。

面试表达：

> 回表是指通过二级索引查到主键值后，再回到主键索引查整行数据。覆盖索引是指查询需要的字段都能从二级索引中直接拿到，不需要回表。减少回表可以通过避免 `select *`、建立合适的联合索引，让 where 条件和 select 字段尽量被同一个索引覆盖。

## 最左前缀原则

联合索引：

```sql
index(a, b, c)
```

底层排序方式：

```text
先按 a 排序
a 相同再按 b 排序
b 相同再按 c 排序
```

能较好用上索引：

```sql
where a = 1
where a = 1 and b = 2
where a = 1 and b = 2 and c = 3
```

不能从头用上这个联合索引：

```sql
where b = 2
where c = 3
where b = 2 and c = 3
```

范围查询后面的列通常难以继续用于索引定位：

```sql
where a = 1 and b > 10 and c = 3
```

通常 `a` 能用，`b` 能做范围，`c` 很难继续用于精确定位。

面试表达：

> 联合索引遵循最左前缀原则。比如索引 `(a,b,c)`，底层是先按 a 排序，a 相同再按 b 排序，b 相同再按 c 排序，所以查询条件要从最左列开始连续匹配，才能有效利用索引。只查 `b` 或 `c` 一般用不上这个联合索引。遇到范围查询后，范围列后面的列通常不能继续用于索引定位。

## 常见索引失效场景

| 场景 | 例子 | 原因 |
|------|------|------|
| 不满足最左前缀 | `where b = 1` | 跳过联合索引最左列 |
| 对索引列做函数 | `where year(create_time)=2026` | B+ 树存的是原始值 |
| 对索引列做表达式 | `where age + 1 = 20` | 破坏索引有序性 |
| 隐式类型转换 | `phone varchar` 却用数字比较 | 可能导致字段转换 |
| 前导模糊匹配 | `like '%tom'` | 无法从前缀定位 |
| 范围查询后继续精确匹配 | `a=1 and b>10 and c=3` | 范围后列难以继续定位 |
| `or` 使用不当 | 一边有索引，一边没索引 | 可能选择全表扫描 |
| 选择性太差 | 查出表里大部分数据 | 优化器可能认为全表扫描更划算 |

可以用：

```sql
where name like 'tom%'
```

尽量避免：

```sql
where name like '%tom%'
```

面试表达：

> 常见索引失效包括：不满足联合索引最左前缀；对索引列使用函数或表达式；发生隐式类型转换；`like` 以 `%` 开头；范围查询后面的列难以继续用于索引定位；`or` 两边索引不完整；以及条件选择性太差，优化器认为全表扫描更划算。实际是否使用索引要看 `EXPLAIN`。

## EXPLAIN 看什么

常看字段：

| 字段 | 重点 |
|------|------|
| `type` | 访问方式，判断是否全表扫描 |
| `possible_keys` | 可能使用哪些索引 |
| `key` | 实际使用了哪个索引 |
| `key_len` | 使用了索引的哪些部分 |
| `rows` | 预估扫描多少行 |
| `Extra` | 额外信息，如覆盖索引、排序、临时表 |

`type` 常见顺序：

```text
const > ref > range > index > ALL
```

`Extra` 常见信息：

| Extra | 含义 |
|-------|------|
| `Using index` | 覆盖索引，通常是好事 |
| `Using where` | 需要额外过滤 |
| `Using filesort` | 额外排序，可能要优化 |
| `Using temporary` | 使用临时表，通常要关注 |

面试表达：

> `EXPLAIN` 我主要看 `type`、`possible_keys`、`key`、`rows` 和 `Extra`。`key` 表示最终使用的索引，`rows` 表示预估扫描行数，`type` 看访问方式，最好不要是 `ALL` 全表扫描。`Extra` 里如果看到 `Using index` 通常说明覆盖索引；如果有 `Using filesort`、`Using temporary`，说明可能有额外排序或临时表，需要进一步优化。

## 慢 SQL 优化思路

常见排查顺序：

1. 通过慢查询日志定位慢 SQL。
2. 用 `EXPLAIN` 看执行计划。
3. 看是否走索引，`key` 是否合理。
4. 看 `rows` 是否很大，是否扫描过多数据。
5. 看是否违反最左前缀或发生索引失效。
6. 减少 `select *`，尝试覆盖索引。
7. 优化 `order by`、`group by`、分页、join。
8. 必要时考虑缓存、拆表、归档历史数据或业务改造。

面试表达：

> 我会先通过慢查询日志定位慢 SQL，再用 `EXPLAIN` 看执行计划，重点关注 `type`、`key`、`rows` 和 `Extra`。如果没有走索引、扫描行数很大、出现 `Using filesort` 或 `Using temporary`，就从索引设计、联合索引顺序、索引失效、回表、排序分页等方向优化。SQL 本身也要避免 `select *`，只查必要字段，尽量用覆盖索引减少回表。必要时再考虑缓存、拆表或业务改造。

## 深分页为什么慢

问题 SQL：

```sql
select *
from user
order by id
limit 1000000, 10;
```

它不是直接跳到第 1000000 行，而是要先扫描或跳过前面 1000000 行，再返回 10 行。

优化方式：

```sql
select *
from user
where id > 1000000
order by id
limit 10;
```

适合按递增主键或稳定游标翻页。

也可以先用覆盖索引查主键，再回表取数据，减少扫描大字段成本。实际是否有效要看执行计划。

面试表达：

> 深分页慢是因为 `limit offset, size` 需要先扫描并跳过 offset 行，再返回 size 行，offset 越大成本越高。优化可以改成基于游标或上次最大 ID 翻页，比如 `where id > last_id order by id limit 10`，让数据库从索引位置继续往后扫。也可以先用覆盖索引查主键，再回表取数据，或者从业务上限制深翻页。

## 容易踩坑的地方

1. 只说 B+ 树“快”，没说核心是减少磁盘 IO。
2. 忘记 InnoDB 主键索引叶子节点存整行数据，二级索引叶子节点存主键值。
3. 能说回表，但说不出覆盖索引如何减少回表。
4. 忘记联合索引按从左到右的顺序组织，跳过最左列通常用不好索引。
5. 只记得“索引会失效”，说不出函数、表达式、隐式转换、前导模糊匹配这些具体场景。
6. `EXPLAIN` 只说“看索引”，说不出 `type`、`key`、`rows`、`Extra`。
7. 慢 SQL 优化只说加索引，忽略回表、排序、分页、扫描行数和业务改造。
8. 不知道深分页慢在需要扫描并丢弃大量 offset 行。

## 我的薄弱点

- B+ 树、聚簇索引、最左前缀、索引失效、深分页这轮基本都是遗忘状态，需要后续复测。
- 回表已经能说出主干：普通索引先查主键 id，再用 id 查整行数据。
- `EXPLAIN` 目前表述偏泛，需要重点记字段：`type`、`key`、`rows`、`Extra`。
- SQL 优化已经能想到看执行计划和避免 `select *`，但还要补全慢查询日志、覆盖索引、排序分页、join、业务改造。
