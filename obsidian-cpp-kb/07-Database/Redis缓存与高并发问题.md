# Redis 缓存与高并发问题

## 一句话理解

Redis 面试重点不是命令大全，而是知道常用数据结构适合什么场景，以及缓存穿透、击穿、雪崩、双写一致性、分布式锁、持久化和淘汰策略怎么处理。

## 常用数据结构

| 类型 | 典型场景 |
|------|----------|
| String | 缓存对象、计数器、分布式锁 |
| Hash | 存对象字段，例如用户信息 |
| List | 简单队列、最新列表 |
| Set | 去重、共同好友、标签集合 |
| ZSet | 排行榜、按分数排序、延迟队列 |

例子：

```text
String:
    user:1 -> json
    count:article:1 -> 100
    lock:order:1 -> uuid

Hash:
    user:1
      name -> Tom
      age  -> 20

List:
    lpush queue task
    rpop queue

Set:
    sadd user:1:tags cpp linux mysql

ZSet:
    zadd rank 100 user1
    zadd rank 90 user2
```

面试表达：

> Redis 常用数据结构有 String、Hash、List、Set、ZSet。String 最常用于缓存、计数器和分布式锁；Hash 适合存对象字段；List 可以做简单队列；Set 适合去重和集合运算；ZSet 带分数排序，适合排行榜、延迟队列等场景。

## 缓存穿透、击穿、雪崩

一句话区分：

```text
穿透：查不存在的数据，缓存没有，数据库也没有
击穿：热点 key 失效，大量请求同时打到数据库
雪崩：大量 key 同时失效，或 Redis 整体不可用
```

| 问题 | 典型原因 | 解决方式 |
|------|----------|----------|
| 缓存穿透 | 查询不存在的数据 | 参数校验、缓存空值、布隆过滤器 |
| 缓存击穿 | 热点 key 过期 | 互斥锁回源、逻辑过期、热点 key 续期 |
| 缓存雪崩 | 大量 key 同时过期或 Redis 故障 | TTL 随机值、Redis 高可用、限流降级、多级缓存 |

缓存穿透：

```text
请求 id=-1 或随机 id
缓存查不到
数据库也查不到
每次都打到数据库
```

缓存击穿：

```text
热点商品 key 过期
大量请求同时进来
全部回源查数据库
```

缓存雪崩：

```text
大量 key 同一时间过期
或 Redis 故障
请求大面积打到数据库
```

面试表达：

> 缓存穿透是查询不存在的数据，缓存和数据库都没有，解决方法有参数校验、缓存空值、布隆过滤器。缓存击穿是热点 key 过期后大量请求同时打到数据库，解决方法是热点 key 逻辑过期、互斥锁回源、异步刷新。缓存雪崩是大量 key 同时失效或 Redis 故障导致请求大面积打到数据库，解决方法是 TTL 加随机值、Redis 高可用、多级缓存、限流降级。

## 缓存和数据库双写一致性

常见推荐：

```text
先更新数据库
再删除缓存
```

不推荐更新数据库后直接更新缓存：

```text
线程 A 更新数据库为 100
线程 B 更新数据库为 200

线程 B 先更新缓存为 200
线程 A 后更新缓存为 100

最后数据库是 200，缓存却是 100
```

不推荐先删缓存再更新数据库：

```text
线程 A 删除缓存
线程 B 发现缓存没了，查数据库旧值 10
线程 B 把旧值 10 写回缓存
线程 A 更新数据库为 20

最后数据库是 20，缓存还是旧值 10
```

推荐流程：

```text
更新数据库成功
删除缓存
下次读缓存未命中
从数据库加载新值
写回缓存
```

注意：这通常保证最终一致性，不是强一致性。

删除缓存失败时需要补偿：

- 删除失败重试。
- 消息队列异步重试。
- 监听 binlog 延迟删除缓存。
- 缓存设置 TTL 兜底。
- 强一致场景考虑读写都走数据库，或使用更严格的同步方案。

面试表达：

> 缓存和数据库双写一般推荐先更新数据库，再删除缓存。因为缓存可以按需重建，删除比更新更不容易写入旧值。先删缓存再更新数据库可能导致并发读把旧数据重新写回缓存；更新数据库后直接更新缓存也可能因为并发写顺序导致旧值覆盖新值。工程里还要处理删除缓存失败，可以用重试、消息队列、binlog 监听和 TTL 兜底，通常保证最终一致性。

## Redis 分布式锁

加锁常用：

```bash
SET lock:order:1 unique_value NX EX 10
```

含义：

- `NX`：key 不存在时才设置，保证只有一个客户端能加锁成功。
- `EX 10`：设置过期时间，防止拿锁进程宕机后锁永远不释放。
- `unique_value`：唯一标识锁持有者，避免误删别人的锁。

不要拆成两条命令：

```bash
SETNX lock 1
EXPIRE lock 10
```

因为 `SETNX` 成功后如果进程崩溃，可能还没来得及设置过期时间，锁就永远不释放。

释放锁不能直接：

```bash
DEL lock
```

锁可能已经过期并被别人重新拿到，直接删会误删别人的锁。

正确思路：

```text
先判断 value 是不是自己的
如果是自己的，再删除
判断和删除要原子执行
通常用 Lua 脚本
```

面试表达：

> Redis 分布式锁通常用 `SET key value NX EX seconds` 实现。`NX` 保证只有 key 不存在时才能加锁成功，`EX` 设置过期时间避免死锁，`value` 用唯一标识区分锁的持有者。释放锁时要先校验 value 是自己的，再删除，并用 Lua 脚本保证校验和删除的原子性，避免误删别人的锁。

## RDB 和 AOF

| 持久化方式 | 核心思路 | 优点 | 缺点 |
|------------|----------|------|------|
| RDB | 定期生成数据快照 | 文件紧凑，适合备份，恢复通常较快 | 可能丢失两次快照之间的数据 |
| AOF | 持续追加写命令日志 | 数据丢失更少，可读性更强 | 文件更大，恢复可能更慢，需要 rewrite |

一句话区分：

```text
RDB：定期生成数据快照
AOF：持续追加写命令日志
```

面试表达：

> RDB 是快照持久化，定期把某一时刻的内存数据保存成文件，适合备份和快速恢复，但可能丢失两次快照之间的数据。AOF 是追加日志，把写命令记录下来，重启时通过重放命令恢复数据，数据安全性通常更好，但文件更大，恢复可能更慢，需要 rewrite 压缩。实际生产中可以结合使用。

## 过期策略和淘汰策略

过期策略回答的是：

```text
设置了 TTL 的 key 什么时候删？
```

Redis 常用：

- 惰性删除：访问 key 时发现过期了，再删除。
- 定期删除：后台定期抽样检查设置了过期时间的 key，发现过期就删。

淘汰策略回答的是：

```text
内存达到 maxmemory 后，选哪些 key 淘汰？
```

常见策略：

| 策略 | 含义 |
|------|------|
| noeviction | 不淘汰，写入报错 |
| allkeys-lru | 从所有 key 中淘汰最近最少使用 |
| volatile-lru | 从设置了过期时间的 key 中淘汰最近最少使用 |
| allkeys-lfu | 从所有 key 中淘汰使用频率低的 |
| volatile-lfu | 从设置了过期时间的 key 中淘汰使用频率低的 |
| volatile-ttl | 优先淘汰更快过期的 key |

面试表达：

> Redis 过期 key 通常通过惰性删除和定期删除结合处理：访问时发现过期就删除，同时后台定期抽样清理过期 key。当内存达到 maxmemory 后，会根据淘汰策略删除 key，比如 noeviction 直接拒绝写入，allkeys-lru 从所有 key 中淘汰最近最少使用，volatile-lru 只从设置了过期时间的 key 中淘汰。纯缓存场景常用 allkeys-lru 或 lfu。

## 容易踩坑的地方

1. 只知道 Redis 快，但说不出各数据结构的典型场景。
2. 混淆缓存穿透、击穿、雪崩：穿透是不存在，击穿是热点过期，雪崩是大面积失效。
3. 双写一致性只说“更新缓存”，忽略并发下旧值覆盖新值。
4. 忘记删除缓存失败需要补偿，不能假设一定删除成功。
5. 分布式锁只说 `SETNX`，忽略设置过期时间必须和加锁原子完成。
6. 释放锁直接 `DEL`，可能误删别人的锁。
7. RDB 和 AOF 只知道名字，说不出快照和命令日志的区别。
8. 混淆过期策略和淘汰策略：过期是 TTL key 怎么删，淘汰是内存满了删谁。
