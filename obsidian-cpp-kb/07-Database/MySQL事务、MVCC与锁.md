# MySQL 事务、MVCC 与锁

## 一句话理解

事务解决“一组操作要不要整体成功”的问题，隔离级别解决“并发事务之间能互相看到什么”的问题，MVCC 用多版本提高读写并发，锁机制负责保护当前读和写操作的正确性。

## ACID

| 特性 | 含义 |
|------|------|
| Atomicity 原子性 | 事务内操作要么全部成功，要么全部失败回滚 |
| Consistency 一致性 | 事务前后数据满足业务约束和完整性约束 |
| Isolation 隔离性 | 并发事务之间不能随意互相干扰 |
| Durability 持久性 | 事务提交后结果要持久保存，宕机也不能丢 |

一致性更像目标，原子性、隔离性、持久性都是帮助实现一致性的手段。

面试表达：

> ACID 分别是原子性、一致性、隔离性、持久性。原子性保证事务内操作要么全成功要么全回滚；一致性保证事务前后数据满足业务和约束规则；隔离性保证并发事务之间不会互相随意影响；持久性保证事务提交后结果即使宕机也不会丢失。

## 隔离级别

| 隔离级别 | 脏读 | 不可重复读 | 幻读 |
|----------|------|------------|------|
| Read Uncommitted 读未提交 | 可能 | 可能 | 可能 |
| Read Committed 读已提交 | 避免 | 可能 | 可能 |
| Repeatable Read 可重复读 | 避免 | 避免 | InnoDB 通过 MVCC + next-key lock 尽量解决 |
| Serializable 串行化 | 避免 | 避免 | 避免 |

MySQL InnoDB 默认隔离级别通常是 `Repeatable Read`。

三个并发问题：

- 脏读：读到了其他事务尚未提交的数据。
- 不可重复读：同一事务内，两次读同一行，内容不一样。
- 幻读：同一事务内，两次范围查询，符合条件的行集合变了。

面试表达：

> MySQL 常见隔离级别有读未提交、读已提交、可重复读、串行化。读未提交可能出现脏读；读已提交解决脏读，但可能不可重复读；可重复读解决不可重复读，InnoDB 默认就是这个级别，并通过 MVCC 和 next-key lock 尽量解决幻读；串行化隔离最强，但并发性能最差。

## MVCC

MVCC 是 Multi-Version Concurrency Control，多版本并发控制。

它的核心作用：

```text
读写并发时，普通快照读不阻塞写，写也不阻塞普通快照读
```

InnoDB 实现 MVCC 主要靠：

- 隐藏字段：记录版本所属事务等信息。
- undo log：保存旧版本数据。
- Read View：判断某个版本对当前事务是否可见。

版本链可以简单理解为：

```text
当前行版本
  -> 上一个旧版本
    -> 更早的旧版本
```

事务读数据时，不是简单读最新值，而是沿着 undo log 版本链找到对自己可见的版本。

面试表达：

> MVCC 是多版本并发控制。InnoDB 通过隐藏字段、undo log 版本链和 Read View，让事务在读取时可以读到符合自己隔离级别的一致性版本。它的主要作用是提高读写并发，让普通快照读不需要加锁，做到读写不互相阻塞。

## 快照读和当前读

快照读：

```sql
select * from user where id = 1;
```

普通 `select` 通常是快照读，读取的是 MVCC 提供的一致性历史版本，不加锁。

当前读：

```sql
select * from user where id = 1 for update;
update user set age = 20 where id = 1;
delete from user where id = 1;
insert into user ...
```

当前读读取最新版本，并且通常会加锁。

面试表达：

> 快照读读取的是 MVCC 提供的一致性历史版本，普通 `select` 通常是快照读，不加锁。当前读读取的是最新版本，并且通常会加锁，比如 `select ... for update`、`update`、`delete`、`insert`。快照读主要解决读写并发问题，当前读用于需要修改或锁定最新数据的场景。

## Read View

Read View 是快照读时生成的可见性视图，用来判断某个数据版本对当前事务是否可见。

简单判断：

```text
版本是已提交的老事务生成的？
    可以看

版本是当前事务自己生成的？
    可以看

版本是当前还没提交的事务生成的？
    不能看，沿 undo log 找旧版本
```

不同隔离级别下 Read View 创建时机不同：

| 隔离级别 | Read View 创建时机 |
|----------|--------------------|
| Read Committed | 每次普通 select 都生成新的 Read View |
| Repeatable Read | 第一次普通 select 生成 Read View，后续复用 |

所以 RC 下可能不可重复读，而 RR 下普通快照读可以可重复读。

面试表达：

> Read View 是 MVCC 在快照读时生成的可见性视图，用来判断某个数据版本对当前事务是否可见。如果当前版本不可见，就沿着 undo log 版本链找更旧的版本。RC 级别下每次 select 生成新的 Read View，RR 级别下通常第一次 select 生成后复用，所以可重复读能保证同一事务内多次快照读结果一致。

## 幻读

幻读关注的是范围查询结果集变化。

```text
第一次查 age > 18，有 10 行
其他事务插入一条 age = 20 并提交
第二次查 age > 18，有 11 行
```

不要和不可重复读混淆：

```text
不可重复读：同一行内容变了
幻读：符合条件的行集合变了
```

InnoDB 在 RR 下：

- 普通快照读：通过 MVCC 和复用 Read View，通常看不到新插入的幻影行。
- 当前读 / 加锁读：通过 next-key lock 锁住范围，防止其他事务插入幻影行。

面试表达：

> 幻读指同一个事务内按相同范围条件查询，两次结果集的行数不一致，通常是其他事务插入或删除了符合条件的记录。InnoDB 在 RR 下，普通快照读通过 MVCC 和复用 Read View 避免看到新插入的数据；当前读则通过 next-key lock，也就是记录锁加间隙锁，锁住范围，防止其他事务插入幻影行。

## 行锁、间隙锁、Next-Key Lock

行锁 / 记录锁：

```text
锁住某条已经存在的索引记录
```

间隙锁：

```text
锁住两个索引记录之间的间隙
防止其他事务在这个范围插入新记录
```

Next-Key Lock：

```text
记录锁 + 间隙锁
```

可以理解成锁住左开右闭区间：

```text
(10, 20]
```

既锁住 `20` 这条记录，也锁住 `(10, 20)` 这个间隙。

面试表达：

> 行锁也叫记录锁，锁的是已经存在的某条索引记录。间隙锁锁的是两个索引记录之间的范围，不锁记录本身，主要防止插入新记录。next-key lock 是记录锁加间隙锁，可以理解为锁住一个左开右闭区间，比如 `(10,20]`，既锁记录也锁间隙，InnoDB 在 RR 下用它来防止当前读出现幻读。

## 行锁和索引

InnoDB 的行锁是基于索引实现的，锁的是扫描到的索引记录。

命中索引：

```sql
update user set age = 20 where id = 10;
```

如果 `id` 是主键索引，可以比较精确地锁住 `id = 10` 对应的记录。

没有命中索引：

```sql
update user set age = 20 where name = 'Tom';
```

如果 `name` 没有索引，可能需要全表扫描，扫描和加锁范围变大，效果上接近锁很多行。

面试表达：

> InnoDB 的行锁是基于索引实现的，锁的是扫描到的索引记录。如果查询条件能命中索引，就可以比较精确地锁定相关记录；如果没有命中索引，可能发生全表扫描，扫描并加锁的范围会变大，导致大量记录被锁住，效果上接近表锁。因此更新、删除、加锁查询时，where 条件最好走索引。

## MySQL 死锁

死锁本质：

```text
事务 A 持有锁 1，等待锁 2
事务 B 持有锁 2，等待锁 1
```

典型场景是多个事务访问资源顺序不一致。

排查：

```sql
show engine innodb status\G
show processlist;
```

重点看：

- 哪两个事务参与死锁。
- 分别执行了什么 SQL。
- 持有什么锁。
- 等什么锁。
- InnoDB 回滚了哪个事务。

处理：

- 应用层捕获死锁错误并重试。
- 多个事务按固定顺序访问资源。
- 缩短事务时间，事务里不做慢操作。
- 确保 `where` 条件走索引，减少锁范围。
- 拆小事务，减少大范围更新。

面试表达：

> MySQL 死锁通常是多个事务加锁顺序不一致，互相持有对方需要的锁，形成循环等待。线上可以用 `show engine innodb status\G` 查看最近一次死锁信息，分析事务 SQL、持有哪些锁、等待哪些锁。处理上，应用层要能捕获死锁并重试，同时优化事务设计，比如固定加锁顺序、缩短事务时间、确保 where 条件走索引、减少大范围更新和锁范围。

## 容易踩坑的地方

1. ACID 字母容易对应错：A 原子性，C 一致性，I 隔离性，D 持久性。
2. 把不可重复读和幻读混在一起；不可重复读看同一行，幻读看范围行集合。
3. 只知道 MVCC 是多版本，但说不出 undo log、版本链、Read View。
4. 不知道 RC 和 RR 的核心差异之一是 Read View 创建时机不同。
5. 把普通 `select` 和 `select ... for update` 都当成同一种读。
6. 把行锁说成锁范围；行锁更准确是锁某条已存在的索引记录。
7. 不知道 next-key lock 是记录锁加间隙锁。
8. 不知道 InnoDB 行锁基于索引实现，没走索引会扩大锁范围。
9. MySQL 死锁只会说“互相等待”，但说不出 `show engine innodb status\G` 排查。
