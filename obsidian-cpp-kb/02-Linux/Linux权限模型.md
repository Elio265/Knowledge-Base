# Linux 权限模型

## 一句话理解

Linux 权限不只是普通文件的读写执行。目录的 `rwx` 含义不同，删除文件本质是修改父目录项；`umask` 是屏蔽权限，特殊权限位用于处理提权、组继承和公共目录删除控制。

## 普通文件权限

| 权限 | 含义 |
|------|------|
| `r` | 读取文件内容 |
| `w` | 修改文件内容 |
| `x` | 把文件作为程序执行 |

权限分三组：

```text
user / group / other
```

例如：

```text
-rw-r--r--
```

表示所有者可读写，同组和其他用户只读。

## 目录权限

| 权限 | 含义 |
|------|------|
| `r` | 读取目录项，能列出目录下的文件名 |
| `w` | 修改目录项，能创建、删除、重命名文件 |
| `x` | 进入目录、路径穿越、访问目录下具体文件 |

目录权限是面试高频坑点：

- 目录只有 `r` 没有 `x`：可能能看到文件名，但不能进入目录，也不能访问文件详细信息。
- 目录只有 `x` 没有 `r`：不能列目录，但如果知道文件名，可以访问里面的文件。

## 删除文件看什么权限

删除文件本质是从父目录里删除“文件名 -> inode”的目录项，所以主要看父目录权限。

删除一个文件需要：

```text
对父目录有 w + x 权限
```

不是看文件本身有没有 `w` 或 `x`。

例子：

```text
dir/
  a.txt
```

删除 `a.txt` 需要对 `dir` 有 `w + x`。即使 `a.txt` 是只读文件，只要父目录允许，也可以删除；反过来，文件可写但父目录不可写，也删不了。

面试表达：

> 删除文件不是修改文件内容，而是修改父目录项，所以看父目录的 `w + x` 权限。

## umask

`umask` 表示创建文件时要屏蔽掉的权限位，不是异或。

计算方式：

```text
最终权限 = 默认权限 & ~umask
```

默认基准通常是：

```text
普通文件: 666
目录:     777
```

如果 `umask = 022`：

```text
文件: 666 & ~022 = 644
目录: 777 & ~022 = 755
```

所以通常会看到：

```bash
touch a.txt   # rw-r--r--，644
mkdir dir     # rwxr-xr-x，755
```

普通文件默认不从 `777` 开始，是因为系统不希望新建普通文件天然可执行。可执行权限应该由用户明确添加：

```bash
chmod +x script.sh
```

## 特殊权限位

| 特殊位 | 作用 | 典型例子 |
|--------|------|----------|
| `setuid` | 可执行文件运行时，进程临时拥有文件所有者权限 | `/usr/bin/passwd` |
| `setgid` | 对可执行文件：运行时拥有文件所属组权限；对目录：新文件继承目录所属组 | 共享项目目录 |
| `sticky bit` | 对目录：只有文件所有者、目录所有者或 root 能删除目录下文件 | `/tmp` |

### setuid

普通用户可以修改自己的密码，是因为 `/usr/bin/passwd` 需要修改 `/etc/shadow`，而普通用户不能直接写这个文件。

`passwd` 通常带 `setuid`：

```text
-rwsr-xr-x root root /usr/bin/passwd
```

用户执行它时，进程临时以文件所有者 root 的权限运行。

### setgid

`setgid` 用在目录上很常见：

```bash
chmod g+s project_dir
```

之后在 `project_dir` 下新建的文件会继承该目录的所属组，适合团队共享目录。

### sticky bit

典型例子是 `/tmp`：

```text
drwxrwxrwt root root /tmp
```

`/tmp` 人人可写，但不能随便删除别人创建的文件。最后的 `t` 就是 sticky bit。

## 容易踩坑的地方

1. 把删除文件理解成修改文件本身：实际是修改父目录项。
2. 删除文件需要父目录 `w + x`，不是文件本身 `w + x`。
3. 目录 `x` 不是执行，而是进入目录和路径穿越。
4. `umask` 不是异或，而是按位屏蔽：`默认权限 & ~umask`。
5. 普通文件默认从 `666` 开始算，目录默认从 `777` 开始算。
6. `setuid` 风险很高，只应该用于非常明确的提权场景。
7. `/tmp` 能人人写但不能互删别人文件，是因为 sticky bit。

## 我的薄弱点

- 删除权限边界：删除文件看父目录 `w + x`，不是文件自身权限。
- `umask` 的计算方式：不是异或，而是屏蔽权限位。
- 特殊权限位不熟：`setuid`、`setgid`、`sticky bit` 需要通过典型场景记忆。

## 成长记录

- 已能准确说出普通权限按 `user/group/other` 三组划分。
- 目录权限、删除权限、`umask` 和特殊权限位仍需后续复测。
