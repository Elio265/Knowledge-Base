# TCP 可靠传输与拥塞控制

## 一句话理解

TCP 的可靠性不是靠业务层“请求应答”，而是靠序列号、ACK、重传、滑动窗口、流量控制和拥塞控制共同保证。

面试先说主干：

> TCP 给字节流编号，接收方用 ACK 表示下一个期望收到的字节序号。发送方根据 ACK 判断哪些数据已经被确认，丢失的数据通过超时重传或快速重传补发；接收方对乱序数据可以缓存，但最终按序交给应用层。发送窗口同时受接收窗口 `rwnd` 和拥塞窗口 `cwnd` 限制。

## TCP 如何保证可靠传输

| 机制 | 作用 |
|------|------|
| 序列号 `SEQ` | 标识每个字节在字节流中的位置 |
| 确认应答 `ACK` | 告诉发送方已经连续收到哪里 |
| 超时重传 | 长时间收不到 ACK 时重发 |
| 快速重传 | 收到多个重复 ACK 时提前重发 |
| 校验和 | 检测数据传输中是否出错 |
| 有序交付 | 乱序数据先缓存，最终按序交给应用层 |
| 滑动窗口 | 允许连续发送多段数据，提高吞吐 |
| 流量控制 | 防止接收方缓冲区被打满 |
| 拥塞控制 | 防止网络被打爆 |

## ACK 确认号是什么意思

`ACK` 确认的是字节序号，不是业务请求编号。

```text
ACK = 1001
```

表示：

```text
1000 及之前的字节已经连续收到
下一次期望收到从 1001 开始的数据
```

普通 ACK 是累计确认：确认的是“到哪个字节为止都连续收到了”，不是只确认某一个包。

## 乱序到达时 ACK 怎么回

假设接收方已经连续收到：

```text
1 ~ 1000
```

但是：

```text
1001 ~ 2000 丢了
2001 ~ 3000 先到了
```

这时接收方仍然回复：

```text
ACK = 1001
```

因为中间缺了 `1001~2000`，连续区间还停在 `1000`。后面的乱序数据可以先缓存，但不能推进普通 ACK，也不能先交给应用层。

## 快速重传和超时重传

如果发送方连续收到多个重复 ACK，例如多个：

```text
ACK = 1001
```

通常说明 `1001` 开始的那段数据可能丢了，但后面的数据还能到达。此时可以触发快速重传，不必等 RTO 超时。

| 机制 | 触发条件 | 含义 |
|------|----------|------|
| 超时重传 | RTO 定时器超时 | ACK 很久没回来，网络可能拥塞较严重 |
| 快速重传 | 通常收到 3 个重复 ACK | 后面的包到了，中间某段大概率丢了 |

重点区别：

- 超时重传是等定时器到期后重发。
- 快速重传是在超时前，根据重复 ACK 提前重发。
- 重复 ACK 不是“超时信号”，而是“缺口信号”。

## 为什么需要滑动窗口

如果每发送一段数据就等一个 ACK，RTT 较大时链路大部分时间都在空等，吞吐很低。

滑动窗口允许发送方在没有收到 ACK 前连续发送多段数据：

```text
已确认 | 已发送未确认 | 可继续发送 | 不可发送
```

当 ACK 推进时，窗口向前滑动，发送方就能继续发送新的数据。

实际可发送的未确认数据量受两个窗口限制：

```text
可发送窗口 = min(rwnd, cwnd)
```

## 流量控制和拥塞控制

| 概念 | 控制对象 | 谁控制 | 作用 |
|------|----------|--------|------|
| `rwnd` 接收窗口 | 接收方缓冲区 | 接收方通告 | 防止把接收方打满 |
| `cwnd` 拥塞窗口 | 网络承载能力 | 发送方维护 | 防止把网络打爆 |

面试表达：

> 流量控制关注接收端处理能力，通过接收方通告的 `rwnd` 限制发送方。拥塞控制关注网络承载能力，由发送方根据丢包、RTT、重复 ACK 等信号动态调整 `cwnd`。

## 慢启动和拥塞避免

慢启动不是增长慢，而是从较小窗口开始探测网络。

```text
cwnd 较小
  -> 慢启动：大致指数增长
  -> 达到 ssthresh
  -> 拥塞避免：线性增长
```

关键点：

- 变的是 `cwnd`，不是接收方的接收能力。
- 慢启动的“慢”指起点小、谨慎探测。
- 达到慢启动阈值 `ssthresh` 后，进入拥塞避免，窗口改为线性增长。

## 丢包后窗口怎么变

常见面试口径可以按 Reno 类模型理解：

| 场景 | 处理 |
|------|------|
| 超时重传 | `ssthresh` 减半，`cwnd` 降到较小值，重新慢启动 |
| 3 次重复 ACK | `ssthresh` 减半，快速重传，进入快速恢复/拥塞避免 |

为什么区别对待：

- 超时重传说明 ACK 长时间没回来，拥塞可能更严重，所以惩罚更重。
- 重复 ACK 说明后续数据还能到达，网络没有完全堵死，所以不需要像超时那样把窗口打回最小。

## 零窗口探测

如果接收方通告：

```text
rwnd = 0
```

表示接收缓冲区满了，发送方不能继续发送普通数据。

但如果接收方后续窗口恢复，而窗口更新 ACK 丢了，双方可能一直等待。为避免死锁，发送方会启动持续计时器，周期性发送零窗口探测报文，询问接收方窗口是否重新打开。

面试表达：

> 当 `rwnd=0` 时，发送方停止发送普通数据，并启动持续计时器，定期发送零窗口探测。接收方如果窗口仍为 0，就继续通告 0；如果窗口打开，就在 ACK 中通告新的窗口大小。

## 容易踩坑的地方

1. 把 TCP 的 ACK 理解成业务请求应答。
2. 把 `ACK=1001` 说成确认“1001 已收到”，正确是“期望下一个从 1001 开始”。
3. 忘记普通 ACK 是累计确认。
4. 把快速重传说成超时重传。
5. 把 `rwnd` 和 `cwnd` 归属说反。
6. 以为慢启动增长慢，实际上前期大致指数增长。
7. 以为快速重传后不降低 `ssthresh`。
8. 把零窗口探测说成普通心跳，面试时最好用准确术语。

## 我的薄弱点

- 一开始把 TCP 可靠性说成“请求应答 + 纠错”，需要改成“序列号、ACK、重传、窗口、控制机制”的体系化表达。
- 术语容易口误：`ACK`、`SEQ` 要和业务请求编号区分开。
- 已理解 ACK 表示下一个期望收到的字节序号，但表达时还要强调“累计确认”。
- 快速重传和超时重传容易混：快速重传由重复 ACK 触发，超时重传由 RTO 触发。
- `rwnd/cwnd` 的归属曾经说反：`rwnd` 是接收方通告，`cwnd` 是发送方根据网络状况维护。
- 慢启动的主干已经掌握，但要避免说成“接收能力指数增长”。
