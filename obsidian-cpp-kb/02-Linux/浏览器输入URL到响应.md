# 浏览器输入 URL 到响应

## 一句话理解

输入 URL 到拿到响应，本质是一条从应用层到链路层再返回的完整链路：浏览器解析 URL，DNS 找 IP，TCP/TLS 建连接，HTTP 发请求，数据经过协议栈封装、路由转发、链路层逐跳发送，服务端处理后再返回响应。

这里按常见 HTTP/1.1 或 HTTP/2 over TCP 的面试链路描述；HTTP/3 基于 QUIC/UDP，需要单独区分。

## 主线流程

```text
1. 浏览器解析 URL，得到协议、域名、端口、路径
2. 查询缓存和 DNS，把域名解析成 IP
3. 根据 IP 和端口建立 TCP 连接
4. 如果是 HTTPS，再进行 TLS 握手
5. 浏览器构造并发送 HTTP 请求
6. 数据经过 TCP 分段、IP 路由、链路层封装发送出去
7. 服务端协议栈逐层解封装，应用层收到 HTTP 请求
8. 服务端处理请求，返回 HTTP 响应
9. 浏览器接收响应，处理状态码、缓存、Cookie，解析渲染页面
10. 连接复用或关闭
```

面试表达：

> 浏览器输入 URL 后，会先解析 URL 并通过 DNS 得到目标 IP，然后和目标 IP、端口建立 TCP 连接；如果是 HTTPS，还会进行 TLS 握手。之后浏览器发送 HTTP 请求，请求经过 TCP/IP 协议栈封装和网络转发到服务端，服务端逐层解封装后交给应用处理，生成 HTTP 响应返回。浏览器收到响应后处理状态码、缓存、Cookie，并解析渲染页面，连接最后可能复用或关闭。

## 协议栈封装与解封装

发送时可以理解为逐层封装：

```text
HTTP 请求
  -> TCP 段
  -> IP 包
  -> 以太网/Wi-Fi 帧
```

接收端反向处理：

```text
链路层解帧
  -> IP 层处理数据包
  -> TCP 层重组字节流
  -> HTTP 层解析请求
  -> 应用程序处理
```

容易混的层次：

| 协议/概念 | 所属层次 | 作用 |
|-----------|----------|------|
| HTTP | 应用层 | 请求/响应语义 |
| TLS | 应用层和传输层之间 | 加密、认证、完整性 |
| TCP | 传输层 | 可靠字节流、端口、连接 |
| IP | 网络层 | 寻址和路由 |
| ARP | 网络层和链路层之间 | 根据下一跳 IP 查 MAC |
| MAC/以太网/Wi-Fi | 链路层 | 当前一跳的数据帧传输 |

## DNS 解析顺序

常见缓存顺序：

```text
浏览器 DNS 缓存
-> 操作系统 DNS 缓存 / hosts
-> 本地 DNS 服务器缓存
```

如果本地 DNS 服务器也没有缓存，一般会从根开始查询：

```text
根 DNS：.com 在哪？
顶级域 DNS：example.com 的权威 DNS 在哪？
权威 DNS：www.example.com 对应的 IP 是多少？
```

关键点：

- 根域在最上面，不是顶级域之后。
- 顶级域是 `.com`、`.cn` 这种。
- `example.com` 这类是二级域。
- 最终给出具体域名 IP 的通常是权威 DNS。

面试表达：

> DNS 解析会先查浏览器缓存、操作系统缓存、hosts 文件和本地 DNS 缓存。如果本地 DNS 没有结果，会从根 DNS 开始迭代查询，根 DNS 返回顶级域 DNS 地址，顶级域 DNS 返回权威 DNS 地址，最后权威 DNS 返回目标域名对应的 IP。本地 DNS 拿到结果后缓存并返回给客户端。

## ARP 查的是下一跳 MAC

DNS 得到的是目标 IP，但链路层真正发送帧时需要目的 MAC。

发送前会先查路由表确定下一跳：

```text
如果目标在同一网段：
    下一跳 = 目标 IP
    ARP 查询目标 IP 的 MAC

如果目标不在同一网段：
    下一跳 = 默认网关 IP
    ARP 查询网关 IP 的 MAC
```

链路层封装时：

```text
目的 MAC = 下一跳 MAC
目的 IP = 最终目标 IP
```

ARP 不负责跨网络转发，跨网络转发靠 IP 路由。ARP 只是在当前局域网内，把下一跳 IP 解析成下一跳 MAC。

## 经过路由器时地址怎么变

普通路由转发时：

| 字段 | 是否变化 | 原因 |
|------|----------|------|
| 源 MAC | 每一跳变化 | 当前发送接口重新封装 |
| 目的 MAC | 每一跳变化 | 指向下一跳设备 |
| 源 IP | 通常不变 | 端到端源地址 |
| 目的 IP | 通常不变 | 端到端目标地址 |

记忆：

> MAC 地址是当前一跳的地址，每经过一跳都会重新封装；IP 地址描述端到端通信，普通路由转发下通常保持不变。

边界：

> 如果经过 NAT，源 IP 或目的 IP 可能会被改写。

## NAT 如何让内网访问公网

NAT 是网络地址转换，常用于让多个私网主机共享一个公网 IP 访问外网，缓解 IPv4 地址不足。

家用或企业网里常见的是 NAPT/PAT，不只改 IP，还会改端口：

```text
内网主机 A：192.168.1.10:50000
NAT 后：公网 IP 1.2.3.4:30001

内网主机 B：192.168.1.11:50000
NAT 后：公网 IP 1.2.3.4:30002
```

NAT 设备维护映射表：

```text
1.2.3.4:30001 -> 192.168.1.10:50000
1.2.3.4:30002 -> 192.168.1.11:50000
```

返回包到达公网 IP 的不同端口后，NAT 设备根据映射表还原成对应的内网 IP 和端口，再转发给内网主机。

## 容易踩坑的地方

1. 把 TCP 说成链路层；TCP 是传输层。
2. 只说“网络层发送”，但没有说清 TCP/IP/链路层逐层封装。
3. DNS 查询顺序说反：应从根 DNS 到顶级域 DNS，再到权威 DNS。
4. 把 ARP 理解成负责跨网络转发；实际跨网络靠 IP 路由。
5. 以为 ARP 查的是最终目标 MAC；跨网段时查的是网关 MAC。
6. 忘记链路层目的 MAC 是下一跳 MAC，IP 包目的 IP 才是最终目标 IP。
7. 忘记路由器转发时 MAC 会变，普通情况下 IP 不变。
8. NAT 不只改 IP，常见场景会改写源 IP 和源端口。

## 我的薄弱点

- 知道 DNS、TCP、网络传输这些环节，但 OSI/TCP-IP 层次映射容易混，曾把 TCP 说到链路层。
- DNS 分级查询顺序需要加强：根 DNS -> 顶级域 DNS -> 权威 DNS。
- ARP 已能放到网络层和链路层之间，但需要记牢“ARP 查下一跳 MAC，不是一路查到最终目标”。
- MAC/IP 变化这点已经明显变扎实：能说出 MAC 每跳变、IP 通常不变。
- NAT 整体思路知道，但表达时要补上“改写源 IP + 源端口”和“端口映射表”。

## 成长记录

- 已能把 DNS 解析、TCP 建连、网络转发放到同一条链路里。
- 已能理解 IP 负责端到端寻址，MAC 负责当前一跳传输。
- 已能回答路由器转发时 MAC 变化、IP 通常不变，并能补充 NAT 例外。
- 下一轮需要复测：协议层次、DNS 查询顺序、ARP 下一跳、NAT 映射表。
