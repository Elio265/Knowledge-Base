# inode 与软硬链接

## 一句话理解

文件名不是文件本体。目录项负责把“文件名”映射到 inode，inode 记录文件元数据和数据块位置；硬链接是多个文件名指向同一个 inode，软链接是一个独立文件，内容是目标路径。

## rm 删除了什么

`rm file` 删除的是目录项，也就是删除“文件名 -> inode”的这条映射，并让 inode 的硬链接计数减 1。

如果文件已经被进程打开：

```text
进程 fd -> struct file -> inode -> 数据块
```

即使路径被 `rm` 删除，已有 fd 仍然有效，进程通常还能继续读写这个文件。只是其他进程不能再通过原路径打开它。

文件真正释放需要同时满足：

1. 硬链接计数为 0。
2. 没有任何打开的 fd / `struct file` 引用它。

这也是为什么删除大日志文件后，磁盘空间可能没有立刻回来：日志进程还持有 deleted 文件的 fd，甚至可能继续写入。

排查命令：

```bash
lsof | grep deleted
ls -l /proc/<pid>/fd
```

可能看到：

```text
3 -> /var/log/app.log (deleted)
```

## 硬链接和软链接

| 对比点 | 硬链接 | 软链接 |
|--------|--------|--------|
| inode | 多个文件名指向同一个 inode | 软链接有自己的 inode |
| 内容 | 共享同一份文件数据 | 内容是目标路径 |
| 删除原路径 | 只要还有硬链接，数据仍在 | 可能变成悬空链接 |
| 跨文件系统 | 通常不支持 | 支持 |
| 链接目录 | 普通用户通常不能创建 | 可以链接目录 |

硬链接：

```text
a.txt ─┐
       ├── same inode -> data blocks
b.txt ─┘
```

软链接：

```text
b.txt -> 自己的 inode -> 内容是 "a.txt" 这个路径
```

命令：

```bash
ln a.txt b.txt      # 硬链接
ln -s a.txt b.txt   # 软链接
```

硬链接没有“原文件”和“链接文件”的主从关系，两个名字地位平等。软链接依赖目标路径，目标路径失效，它就失效。

## 为什么硬链接不能跨文件系统

inode 不是全局编号，只在自己的文件系统内部有意义。

```text
文件系统 A:
  inode 100 -> /a.txt

文件系统 B:
  inode 100 -> /b.txt
```

这两个 `inode 100` 完全不是同一个对象。真正定位一个文件，至少需要：

```text
设备号 / 文件系统 + inode 号
```

硬链接的本质是在目录中增加一个名字，让它指向同一个 inode。文件系统 B 的目录项只能指向自己管理的 inode，不能直接指向文件系统 A 管理的 inode。

更重要的是，硬链接还要维护 inode 链接计数。如果允许跨文件系统硬链接，两个独立文件系统就要共同维护链接计数、空间释放、崩溃恢复、卸载和 fsck 一致性，文件系统边界会被破坏。

软链接可以跨文件系统，因为它只是一个独立文件，内容是路径字符串：

```text
softlink -> "/mnt/diskA/a.txt"
```

访问时再按路径重新查找目标。

## 为什么普通用户不能硬链接目录

目录硬链接会破坏目录树结构，形成环。

```text
/a/b -> /a
```

这样目录结构就不再是树，而是图。`find`、`tree`、备份、递归删除都可能陷入循环，目录链接计数和一致性维护也会变复杂。

目录里的 `.` 和 `..` 本质上也和目录链接有关，但它们由文件系统严格维护，不允许普通用户随意创建。

## 容易踩坑的地方

1. `rm` 删除的是目录项，不是立刻清除 inode 和数据块。
2. 已经打开的文件被删除后，进程仍可通过原 fd 继续读写。
3. 删除大文件后空间没释放，常见原因是进程仍持有 deleted 文件 fd。
4. 硬链接共享 inode，没有主从关系。
5. 软链接有自己的 inode，内容是目标路径，目标路径失效就会悬空。
6. inode 只在同一文件系统内有意义，所以硬链接通常不能跨文件系统。
7. 普通用户不能硬链接目录，核心原因是避免目录环和一致性问题。

## 我的薄弱点

- 删除路径名和关闭打开文件对象是两件事：路径没了，已打开 fd 仍可能读写。
- 磁盘空间释放条件：链接计数为 0，并且没有打开文件对象引用。
- 跨文件系统硬链接的本质限制：inode 不是全局对象，链接计数也不能跨文件系统维护。
