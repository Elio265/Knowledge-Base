# 进程与线程

## 一句话理解

进程是资源分配的基本单位，线程是 CPU 调度的基本单位。Linux 中线程本质上也是 `task`，可以理解为共享部分资源的轻量级进程。

## 核心结论

1. 进程之间地址空间隔离，通信通常需要管道、消息队列、共享内存、Socket 等 IPC 机制。
2. 同一进程内的线程共享地址空间和部分进程资源，因此通信更方便，但必须处理同步问题。
3. Linux 内核调度的对象是 task；线程通常通过 `clone()` 创建，通过共享地址空间、文件描述符等资源表现为同一进程内的多个执行流。
4. `fork()` 后父子进程拥有独立的虚拟地址空间，内存通过写时拷贝优化；文件描述符表被复制，但底层打开文件对象可能共享。

## 线程共享和独立资源

| 类别 | 内容 |
|------|------|
| 线程间共享 | 地址空间、代码段、数据段、堆、全局变量、静态变量、文件描述符表、当前工作目录、信号处理方式 |
| 线程独立 | 线程栈、寄存器上下文、程序计数器、线程 ID、调度状态、`errno`、信号屏蔽字 |

注意：线程栈虽然每个线程独立使用，但仍位于同一个进程地址空间中。如果一个线程拿到另一个线程栈变量的地址，理论上也能访问，所以多线程代码更要注意生命周期和同步。

## 文件描述符的多线程坑

同一进程内多个线程共享文件描述符表。如果线程 A 调用：

```c
close(fd);
```

线程 B 不能继续把这个 `fd` 当作原来的文件或 Socket 使用。

原因是 `fd` 只是文件描述符表中的编号，`close(fd)` 会移除这个表项。之后线程 B 再使用它，可能得到 `EBADF`，也可能因为该编号被新的 `open/accept/socket` 复用而误操作另一个对象。

如果线程 B 已经阻塞在 `read(fd)` 上，线程 A 调用 `close(fd)` 不一定能立刻唤醒 B。阻塞中的系统调用可能已经持有底层文件对象引用，B 可能在有数据、EOF、信号中断或错误发生后才返回。

Socket 场景下，如果想主动打断读写，更常见的做法是使用：

```c
shutdown(fd, SHUT_RDWR);
```

或者使用 `eventfd`、管道、自唤醒机制、超时和非阻塞 IO 做退出控制。

## fork 后父子进程关系

| 内容 | 父子进程关系 |
|------|--------------|
| 进程地址空间 | 逻辑上独立，刚开始通过写时拷贝共享物理页 |
| PCB / `task_struct` | 独立，各自有 PID、调度状态 |
| 文件描述符表 | 各自有一份表，但表项指向同一个底层打开文件对象 |
| 文件偏移量 | 共享，因为偏移量在底层打开文件对象中 |
| 信号处理函数 | 子进程继承父进程设置，之后不是实时共享 |
| 信号屏蔽字 | 子进程继承一份副本，之后各改各的 |
| 未决信号 | 子进程通常不继承父进程的未决信号 |

重点：`fork()` 不是让父子进程共享所有资源，而是复制进程上下文；内存靠写时拷贝优化，文件描述符复制表项但共享底层打开文件对象。

## 写时拷贝

`fork()` 后父子进程的全局变量不是逻辑共享的。

刚开始父子进程可能映射到同一批只读物理页；当其中一方写入全局变量、堆或栈上的数据时，内核会触发写时拷贝，为写入方复制新的物理页。之后父子进程修改的就是各自的内存，不会互相影响。

面试表达：

> 虚拟地址可能相同，不代表是同一个变量实体。父子进程拥有独立虚拟地址空间，写时拷贝只是内核为了减少 `fork()` 成本做的优化。

## 容易踩坑的地方

1. 把“线程直接通信”说得太绝对：线程共享地址空间，可以直接读写共享变量，但必须配合同步机制。
2. 混淆信号处理方式和信号屏蔽字：信号处理方式线程间共享，信号屏蔽字每个线程独立。
3. 以为 `close(fd)` 一定能唤醒另一个线程中阻塞的 `read(fd)`：不一定。
4. 混淆 `fork()` 后的 fd 表和底层打开文件对象：fd 表复制，底层 open file object 可能共享。
5. 以为父子进程全局变量共享：逻辑上不共享，只是写之前可能共享物理页。

## 我的薄弱点

- 线程共享资源和独立资源的边界，尤其是信号处理方式与信号屏蔽字。
- 多线程下文件描述符关闭、复用和阻塞读写之间的关系。
- `fork()` 后“复制一份”和“共享底层对象”的区别，尤其是文件描述符表和文件偏移量。
