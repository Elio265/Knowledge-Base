# TCP 与 UDP

## 一句话理解

TCP 适合可靠、有序、完整的数据传输；UDP 适合低延迟、实时性强、能容忍一定丢包的场景。

面试先说主干：

> TCP 是面向连接、可靠、有序、面向字节流的传输层协议，有 ACK、重传、滑动窗口、流量控制和拥塞控制。UDP 是无连接、不保证可靠、不保证有序、面向数据报的传输层协议，开销更小、延迟更低，但可靠性需要应用层自己取舍或补充。

## TCP 和 UDP 的区别

| 对比 | TCP | UDP |
|------|-----|-----|
| 连接 | 面向连接，需要三次握手 | 无连接，发送前不需要建立连接 |
| 可靠性 | 可靠传输，有 ACK、重传、序列号 | 不保证可靠，可能丢包 |
| 顺序 | 保证有序交付 | 不保证有序 |
| 数据边界 | 面向字节流，没有消息边界 | 面向数据报，保留消息边界 |
| 控制机制 | 有流量控制和拥塞控制 | 没有内置拥塞控制 |
| 开销 | 头部和状态维护较重 | 头部小，状态少 |
| 场景 | HTTP/HTTPS、文件传输、数据库、SSH | DNS、直播、语音视频、游戏、QUIC/HTTP3 |

## 字节流和数据报

TCP 面向字节流，应用层看到的是一串连续字节，没有天然消息边界。

```text
send("hello")
send("world")

recv 可能读到：
"hell"
"oworld"
或者 "helloworld"
```

所以 TCP 应用层要自己设计协议边界，例如长度字段、分隔符，处理粘包和半包。

UDP 面向数据报，一次 `sendto` 发送一个 datagram，接收端一次 `recvfrom` 通常读取一个 datagram，消息边界被保留。

```text
sendto("hello")
sendto("world")

recvfrom 通常读到：
"hello"
下一次读到：
"world"
```

注意：

> UDP 接收缓冲区如果装不下一个完整数据报，超出的部分可能被截断丢弃，不会像 TCP 那样下次继续读剩余字节。

## 为什么实时场景常用 UDP

DNS、直播、语音、游戏等场景常用 UDP，核心原因是：

- 更看重低延迟和实时性。
- 能容忍少量丢包。
- 旧数据迟到后价值不大。
- 应用层可以自己决定是否重传、如何纠错、如何丢弃过期数据。

例子：

```text
直播/语音：
    丢一帧可以跳过，迟到的旧数据价值不大

游戏：
    位置状态频繁更新，旧状态丢了可以被新状态覆盖

DNS：
    请求和响应通常较小，失败可以重试
```

UDP 不一定总比 TCP 快。实际性能取决于网络环境、应用层协议设计和可靠性需求。如果应用层在 UDP 上又实现大量确认、重传和拥塞控制，开销也会上升。

不要说：

> TCP 必须每个包等 ACK 才能发下一个包。

更准确是：

> TCP 有滑动窗口，可以连续发送多个未确认数据段，但可靠传输、有序交付、重传和拥塞控制会带来额外开销和延迟。

## 基于 UDP 实现可靠传输

UDP 本身不保证可靠。如果要基于 UDP 做可靠传输，需要在应用层补机制：

| 机制 | 作用 |
|------|------|
| 序列号 | 标识每个数据包，识别乱序和重复 |
| ACK 确认 | 告诉发送方哪些包收到了 |
| 超时重传 | 长时间没收到确认就重发 |
| 滑动窗口 | 不用发一个等一个，提高吞吐 |
| 乱序缓存 | 后到的包先缓存，缺口补齐后再交付 |
| 去重 | 重传包可能重复到达，需要过滤 |
| 流量控制 | 防止接收方缓冲区被打满 |
| 拥塞控制 | 防止把网络打爆 |
| 校验 | 检测数据损坏 |

不一定非要逐个 ACK，也可以累计 ACK、选择性 ACK 或批量确认，取决于协议设计。

面试表达：

> UDP 本身不保证可靠，如果要实现可靠传输，需要在应用层补序列号、ACK、超时重传、滑动窗口、乱序缓存、去重、流量控制和拥塞控制等机制。这样能按业务需要实现可靠、有序或部分可靠传输，但复杂度和开销也会上升。

## QUIC 为什么基于 UDP

HTTP/2 虽然能在一条 TCP 连接上做多个 stream 的多路复用，但 TCP 底层仍然是一个有序字节流。

问题是：

```text
stream A 的某个 TCP 包丢了
TCP 必须等这个缺口重传补齐
后面的字节即使属于 stream B，也不能先交给上层
```

所以 HTTP/2 解决了 HTTP 应用层的队头阻塞，但没有解决 TCP 层的队头阻塞。

QUIC 基于 UDP，在应用层自己实现可靠传输和多路复用：

```text
stream A 丢包
只阻塞 stream A
stream B / stream C 可以继续交付
```

QUIC 选择 UDP 的常见原因：

- 缓解 TCP 层队头阻塞。
- 集成 TLS 1.3，减少握手延迟。
- 在用户态实现，协议升级更灵活。
- 支持连接迁移，例如 Wi-Fi 切换到 4G 时可以通过连接 ID 维持连接。

面试表达：

> QUIC 基于 UDP，是因为 TCP 天然是有序字节流。HTTP/2 虽然做了多路复用，但底层 TCP 一旦丢包，所有 stream 都会受阻。QUIC 在 UDP 之上自己实现可靠传输和多路复用，使丢包只影响对应的 stream，减少队头阻塞。同时 QUIC 集成 TLS 1.3，减少握手延迟，并且在用户态实现，方便协议演进和连接迁移。

## 容易踩坑的地方

1. 只背“TCP 可靠、UDP 不可靠”，没有说出连接、有序、字节流、数据报、拥塞控制和场景。
2. 忘记 UDP 保留消息边界，TCP 没有消息边界。
3. 以为 UDP 一定比 TCP 快；实际取决于业务和应用层协议设计。
4. 说 TCP 每个包都要等 ACK 才发下一个；实际 TCP 有滑动窗口。
5. 以为 UDP 可靠传输只要 ACK 和重传，忘了序列号、乱序、去重、窗口和拥塞控制。
6. 只说 QUIC 用 UDP 是因为 UDP 快，漏掉 TCP 层队头阻塞、多路复用、握手延迟和用户态演进。

## 我的薄弱点

- TCP/UDP 基础定义能说出来，但一开始偏背定义，还需要主动补维度：有序性、数据边界、控制机制和典型场景。
- 字节流和数据报的主干能理解，但要继续练“对应用层读取的影响”。
- 对 UDP 实时场景理解较好，知道能容忍少量丢包、实时性优先。
- 表达 TCP 开销时要避免说“每个包都要等 ACK 才发下一个”，应改成“TCP 有滑动窗口，但可靠机制有额外开销”。
- QUIC 的关键点已经能抓到 TCP 队头阻塞，但要补充 HTTP/2 只解决应用层队头阻塞，QUIC 缓解 TCP 层队头阻塞。

## 成长记录

- 已能准确说出 TCP 是可靠、有连接、面向字节流，UDP 是无连接、不可靠、面向数据报。
- 已能理解 UDP 适合直播、游戏、DNS 等低延迟场景。
- 已经意识到 UDP 不一定绝对比 TCP 快，需要看应用层是否补可靠机制。
- 已能说出 QUIC 使用 UDP 的核心原因之一是缓解 TCP 字节流队头阻塞。
