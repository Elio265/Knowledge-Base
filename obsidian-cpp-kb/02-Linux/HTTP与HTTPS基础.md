# HTTP 与 HTTPS 基础

## 一句话理解

HTTP 是明文应用层协议；HTTPS 是在 HTTP 和 TCP 之间加入 TLS，提供加密、身份认证和完整性保护。

面试表达：

> HTTP 默认端口是 80，数据明文传输，可能被窃听、篡改或伪装。HTTPS 默认端口是 443，本质是 HTTP + TLS，TLS 通过证书验证服务器身份，通过密钥协商得到会话密钥，后续使用对称加密传输数据，并校验数据完整性。

## HTTP 和 HTTPS 的区别

| 对比 | HTTP | HTTPS |
|------|------|-------|
| 安全性 | 明文传输 | TLS 加密传输 |
| 默认端口 | 80 | 443 |
| 身份认证 | 无内置认证 | 通过证书验证服务器身份 |
| 完整性 | 容易被篡改 | 可检测篡改 |
| 性能 | 少一次 TLS 握手 | 有 TLS 握手和加密开销 |

不要说“TLS 的 SSL 加密”。更准确说：

```text
HTTPS = HTTP + TLS
```

SSL 是早期名称和旧协议，面试里直接说 TLS 更稳。

## HTTPS 为什么更安全

HTTPS 主要解决三个问题：

| 能力 | 作用 |
|------|------|
| 加密 | 防止内容被窃听 |
| 认证 | 通过证书验证服务器身份，防止中间人伪装 |
| 完整性 | 防止传输内容被篡改 |

## 为什么要混合加密

HTTPS 不全程使用非对称加密，因为非对称加密计算成本高，不适合大量业务数据传输。

更常见的做法：

```text
握手阶段：使用非对称加密或密钥交换算法协商会话密钥
传输阶段：使用会话密钥做对称加密
```

对称加密性能更好，适合加密后续 HTTP 数据；难点是双方要安全地拿到同一个会话密钥，所以 TLS 在握手阶段先完成密钥协商。

面试简版：

> 非对称加密用于身份认证和密钥协商，对称加密用于后续数据传输。

## 证书如何防止中间人攻击

浏览器信任的不是网站自己说“我是真的”，而是信任 CA 机构签发的证书链。

验证流程可以简化为：

```text
服务器发送证书
浏览器用内置可信根 CA 验证证书链和签名
检查域名是否匹配
检查证书是否过期或被吊销
验证通过后，才相信证书里的公钥属于目标服务器
```

中间人可以自己生成假证书，但没有可信 CA 的私钥，无法给假证书生成合法签名，所以浏览器验证不过。

核心关键词：

```text
CA 签名 + 证书链校验 + 域名校验 + 有效期/吊销检查
```

## TLS 握手做了什么

不用死背字段，抓住四件事：

```text
协商参数
验证身份
协商会话密钥
切换到加密通信
```

简化流程：

1. 客户端发送 `ClientHello`，告诉服务端支持的 TLS 版本、加密套件，并带上随机数。
2. 服务端返回 `ServerHello`，选择版本和加密套件，并发送服务器证书。
3. 客户端验证证书是否合法。
4. 双方通过密钥交换算法协商出会话密钥。
5. 握手完成后，后续 HTTP 数据使用会话密钥对称加密传输。

面试表达：

> TLS 握手主要完成加密参数协商、服务器身份验证和会话密钥协商。握手完成后，HTTP 数据通过对称加密传输。

## HTTP 无状态是什么意思

HTTP 无状态是指协议本身不会记住上一次请求是谁发的、做了什么。每次请求在协议层面都是独立的。

例如：

```text
第一次请求：登录成功
第二次请求：访问购物车
```

HTTP 协议本身不会自动知道第二次请求就是刚才登录的用户，所以需要 Cookie / Session 等机制维护用户状态。

## Cookie 和 Session

| 对比 | Cookie | Session |
|------|--------|---------|
| 保存位置 | 客户端浏览器 | 服务端 |
| 保存内容 | 一小段键值数据，常见是 `session_id` | 用户登录状态、用户信息等上下文 |
| 安全性 | 客户端可见，可能被窃取或篡改 | 状态在服务端，相对更可控 |
| 请求行为 | 浏览器会随请求自动携带 | 服务端根据 `session_id` 查找 |

典型流程：

```text
用户登录成功
服务端创建 Session
服务端通过 Set-Cookie 返回 session_id
浏览器保存 Cookie
后续请求自动带上 session_id
服务端根据 session_id 找到用户上下文
```

面试表达：

> Cookie 存在客户端，Session 存在服务端。常见做法是 Cookie 保存 `session_id`，服务端根据 `session_id` 查找对应 Session，从而在无状态 HTTP 上识别用户身份和会话上下文。

## 用户信息直接放 Cookie 的风险

- Cookie 存在客户端，敏感信息明文放进去容易泄露。
- 如果没有签名或校验，用户可能篡改 Cookie，例如把 `role=user` 改成 `role=admin`。
- Cookie 会随请求发送，放太多数据会增加请求开销。
- 如果没有 HTTPS、`HttpOnly`、`Secure`、`SameSite` 等保护，可能被中间人、XSS 或 CSRF 利用。

原则：

> Cookie 里不要直接明文保存敏感信息。需要保存状态时，优先只放标识或 token，并配合签名、加密和安全属性。

## 常见状态码

| 状态码 | 含义 |
|--------|------|
| `100 Continue` | 客户端可以继续发送请求体 |
| `200 OK` | 请求成功 |
| `206 Partial Content` | 返回部分内容，常见于 Range 请求、断点续传、视频拖动 |
| `301 Moved Permanently` | 永久重定向 |
| `302 Found` | 临时重定向 |
| `304 Not Modified` | 资源未修改，客户端继续使用缓存 |
| `400 Bad Request` | 请求格式或参数错误 |
| `401 Unauthorized` | 未认证，需要登录或认证信息无效 |
| `403 Forbidden` | 已认证或已理解请求，但没有权限 |
| `404 Not Found` | 资源不存在 |
| `500 Internal Server Error` | 服务端内部错误 |
| `502 Bad Gateway` | 网关或代理从上游收到错误响应 |
| `503 Service Unavailable` | 服务暂时不可用，可能过载或维护 |

容易混的：

```text
401：你是谁还没确认，或者认证失败
403：知道你是谁，但你没权限
```

## GET 和 POST

| 对比 | GET | POST |
|------|-----|------|
| 语义 | 获取资源 | 提交数据 |
| 参数位置 | 通常放在 URL 查询字符串 | 通常放在请求体 body |
| 幂等性 | 一般幂等 | 通常不幂等 |
| 缓存 | 通常可缓存 | 默认不缓存 |
| 浏览器行为 | 可被历史记录、书签、日志记录 | 参数不直接出现在 URL 中 |

常见误区：

> GET 和 POST 是否明文，取决于使用 HTTP 还是 HTTPS，不取决于请求方法。

```text
HTTP 下：GET / POST 都是明文
HTTPS 下：GET / POST 的传输内容都会被 TLS 加密
```

GET 参数放在 URL 里，更容易出现在浏览器历史记录、服务器访问日志、代理日志和书签中，所以不适合放敏感信息。POST 参数放在 body 中，但如果走 HTTP，也同样是明文传输。

## 幂等性

幂等是指：

> 同一个请求执行一次和执行多次，对服务端资源状态的影响相同。

注意：幂等不是说每次返回内容一定完全一样，而是说对资源状态的影响一样。

例子：

```text
GET /user/1
```

通常只是查询资源，请求一次和多次不会改变资源状态，所以一般是幂等的。

```text
POST /orders
```

可能每请求一次就创建一个新订单，请求两次就创建两个订单，所以通常不幂等。

补充：

- GET 语义上不应修改资源，但可能会记录访问日志、统计数据，这不影响资源语义。
- POST 也可以通过幂等键等机制做成幂等，但语义上通常不默认幂等。

## HTTP 长连接和 keep-alive

HTTP 长连接是指多个 HTTP 请求复用同一条 TCP 连接，避免每次请求都重新三次握手和四次挥手。

`Connection: keep-alive` 的含义是：

```text
这次 HTTP 请求/响应结束后，不要立刻关闭底层 TCP 连接
后续请求可以继续复用这条连接
```

它解决的问题：

- 减少频繁建立和关闭 TCP 连接的开销。
- 降低请求延迟。
- 对 HTTPS 来说，还可以减少 TLS 握手成本。

不要把 HTTP keep-alive 和 TCP keepalive 混在一起：

| 概念 | 所属层次 | 作用 |
|------|----------|------|
| HTTP keep-alive | 应用层 | 复用 TCP 连接发送多个 HTTP 请求 |
| TCP keepalive | 传输层 | 长时间空闲后探测对端是否还活着 |

## HTTP/1.1 和 HTTP/2 多路复用

HTTP/1.1 支持长连接，可以复用同一条 TCP 连接发送多个请求，但主要问题是并发能力有限，容易受到队头阻塞影响。

简化理解：

```text
HTTP/1.1：
一个 TCP 连接上复用多个请求
但请求/响应处理基本偏串行
前一个响应慢，后面的响应容易被挡住
```

HTTP/2 引入二进制分帧和多路复用：

```text
一个 TCP 连接
  -> 多个 stream
  -> 请求/响应拆成 frame
  -> 不同 stream 的 frame 可以交错传输
```

所以 HTTP/2 能在一条 TCP 连接上并发处理多个请求，减少连接数量，提高并发效率。

但要注意：

> HTTP/2 解决了 HTTP 应用层的队头阻塞，但它底层仍基于 TCP。TCP 一旦丢包，后续字节仍要等待重传，所以仍可能有 TCP 层面的队头阻塞。

## HTTP 缓存

HTTP 缓存可以按两类记：

```text
强缓存：
    Cache-Control
    Expires

协商缓存：
    ETag / If-None-Match
    Last-Modified / If-Modified-Since
```

强缓存命中时，浏览器直接使用本地缓存，不向服务器发送请求。

协商缓存会向服务器发送条件请求：

```text
资源没变：返回 304 Not Modified，继续用本地缓存
资源变了：返回 200 和新资源
```

常见字段：

| 字段 | 作用 |
|------|------|
| `Cache-Control` | 控制缓存策略，例如 `max-age`、`no-cache`、`no-store` |
| `Expires` | 缓存过期的绝对时间，受客户端本地时间影响 |
| `ETag` | 资源版本标识，用于判断资源内容是否变化 |
| `Last-Modified` | 资源最后修改时间 |
| `If-None-Match` | 请求时带上旧 `ETag`，让服务器判断资源是否变化 |
| `If-Modified-Since` | 请求时带上旧修改时间，询问资源是否变化 |

常见优先级：

```text
Cache-Control > Expires
ETag > Last-Modified
```

## 容易踩坑的地方

1. 只说 HTTPS 加密，漏掉身份认证和完整性校验。
2. 把 SSL 和 TLS 混着说，面试里优先说 TLS。
3. 以为 HTTPS 全程使用非对称加密。
4. 只知道证书来自 CA，但说不出证书链、签名、域名、有效期校验。
5. 不熟 TLS 握手流程。
6. HTTP 无状态表达太绕，应该说“请求之间默认独立，协议不保存上下文”。
7. Cookie / Session 知道协作流程，但容易漏掉安全风险。
8. 状态码具体语义容易混，尤其是 `206`、`304`、`401/403`、`502/503`。
9. 以为 GET 明文、POST 不明文；实际是否加密取决于 HTTP/HTTPS。
10. 幂等不是“返回结果完全一样”，而是“对资源状态影响一样”。
11. 把 HTTP keep-alive 连接复用和 TCP keepalive 探活混在一起。
12. 以为 HTTP/1.1 长连接不能复用；实际是能复用，但并发复用能力有限。
13. HTTP/2 解决的是应用层队头阻塞，底层 TCP 丢包仍会阻塞后续字节。
14. HTTP 缓存要放进“强缓存 / 协商缓存”模型里理解，不要只背字段名。

## 我的薄弱点

- HTTPS 基础区别知道，但一开始只说了“明文/加密”和端口，漏了认证、完整性和证书体系。
- “TLS / SSL” 表达需要修正：现代口径优先说 TLS。
- 混合加密理解较好，已经能说出非对称加密成本高、对称加密适合数据传输。
- CA 方向知道，但术语需要说准：CA、证书链、数字签名、浏览器根证书。
- TLS 握手目前属于概念空白，需要后续复测。
- Cookie / Session 的协作流程已经能说出来，但 Cookie 安全风险需要主动补充。
- HTTP 状态码大类清楚，具体语义要复测，尤其是 `401/403` 和 `502/503`。
- GET / POST 主干知道，但把 GET 和“明文传输”绑定了，需要改成“是否加密取决于 HTTP/HTTPS”。
- 幂等性已经能说出主干，但要强调“对资源状态的影响”，不是返回内容。
- HTTP 长连接理解到“复用 TCP 连接”，但把 HTTP keep-alive 和 TCP keepalive 混淆了。
- HTTP/2 的多路复用理解还不稳：HTTP/1.1 不是不能复用，而是不能很好地并发复用。
- HTTP 缓存字段知道大概作用，但还没形成强缓存 / 协商缓存的整体模型。
