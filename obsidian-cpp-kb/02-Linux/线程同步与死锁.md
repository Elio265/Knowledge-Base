# 线程同步与死锁

## 一句话理解

线程同步的核心是控制多个线程访问共享资源的顺序。互斥锁保护临界区，条件变量负责等待和通知，信号量控制资源数量，原子操作适合单变量的简单并发读改写，线程池复用工作线程，死锁则是多个线程互相等待资源导致无法继续推进。

## 互斥锁、自旋锁、读写锁

| 类型 | 等待方式 | 特点 | 适用场景 |
|------|----------|------|----------|
| 互斥锁 | 拿不到锁通常阻塞/睡眠 | 让出 CPU，避免忙等 | 临界区稍长、竞争明显 |
| 自旋锁 | 拿不到锁一直循环尝试 | 不切换线程，但会占 CPU | 临界区极短、锁很快释放 |
| 读写锁 | 读共享，写独占 | 多个读者并发，写者独占 | 读多写少 |

面试表达：

> 互斥锁用于保护临界区，多个线程竞争同一把锁，拿不到锁时通常会阻塞，让出 CPU。自旋锁拿不到锁时不会睡眠，而是一直循环尝试获取锁，适合临界区很短、锁很快释放的场景。读写锁把访问分为读锁和写锁，多个读者可以并发，写者需要独占，适合读多写少的场景。

注意：

- 互斥锁更常见于多个线程竞争同一把锁；进程间也可以用进程共享互斥锁，但不是默认重点。
- 读写锁不是“读一定无锁”，而是多个读者可以同时持有读锁；写锁仍然独占。

## 自旋锁为什么不能乱用

自旋锁拿不到锁时会一直忙等，占用 CPU。

如果临界区很长：

```text
锁很久不释放
等待线程一直自旋
CPU 被无效消耗
```

单核 CPU 上更不适合：

```text
线程 A 持有锁
线程 B 在唯一 CPU 上自旋等待
线程 A 没机会运行
线程 A 也就没机会释放锁
```

面试表达：

> 自旋锁适合临界区极短的场景。如果临界区很长，自旋线程会长时间空转浪费 CPU。单核 CPU 上更不适合，因为等待线程可能占据唯一 CPU，使持锁线程没有机会运行并释放锁。

## 条件变量

条件变量用于线程间等待和通知，常见于生产者消费者模型。

它解决的是：

```text
条件不满足时，让线程睡眠等待
条件满足后，由其他线程通知唤醒
```

条件变量通常要和互斥锁一起使用：

- 互斥锁保护共享状态，例如队列是否为空。
- 条件变量负责让线程在条件不满足时睡眠，避免一直轮询。

`wait` 的关键动作：

```text
原子地释放互斥锁
进入等待队列睡眠
被唤醒后，重新获取互斥锁
再次检查条件
```

典型写法：

```cpp
std::unique_lock<std::mutex> lock(mtx);
while (queue.empty()) {
    cv.wait(lock);
}
auto task = queue.front();
queue.pop();
```

生产者：

```cpp
{
    std::lock_guard<std::mutex> lock(mtx);
    queue.push(task);
}
cv.notify_one();
```

必须用 `while` 判断条件，而不是只用 `if`：

- 可能发生虚假唤醒。
- 多个线程被唤醒后，条件可能已经被其他线程先消费。

面试表达：

> 条件变量用于线程间等待和通知。线程在条件不满足时调用 `wait`，它会原子地释放互斥锁并进入睡眠；被 `notify` 唤醒后，会先重新获取互斥锁，再继续检查条件。条件变量必须配合互斥锁保护共享状态，并且等待条件要用 `while` 循环判断，防止虚假唤醒或被其他线程抢先消费。

## 信号量

信号量本质是一个计数器，用来表示当前还有多少个资源可用。

```text
P / wait:
    获取资源，计数减 1
    如果计数为 0，就阻塞等待

V / post:
    释放资源，计数加 1
    可能唤醒等待线程
```

信号量和互斥锁的区别：

| 对比 | 互斥锁 | 信号量 |
|------|--------|--------|
| 核心作用 | 保护临界区 | 控制资源数量 |
| 同时进入数量 | 通常只允许 1 个线程 | 可以允许 N 个线程 |
| 所有权 | 强调谁加锁谁解锁 | 不强调严格所有权 |
| 场景 | 保护共享变量、队列、map | 连接池、资源池、生产者消费者 |

二值信号量和互斥锁效果上相似，但不完全等价：

- 二值信号量计数只有 `0/1`，可以用于实现互斥。
- 互斥锁有锁拥有者概念，通常要求加锁线程自己解锁。
- 信号量更偏资源计数和线程同步。

面试表达：

> 信号量本质是一个资源计数器，用于控制同时可访问资源的数量。互斥锁主要用于保护临界区，强调同一时间只有一个线程进入，并且通常要求谁加锁谁解锁。二值信号量计数只有 0/1，在效果上可以实现互斥，但它不完全等同于互斥锁，因为信号量更偏资源计数和线程同步，不强调锁拥有者。

## 有界生产者消费者

有界队列通常需要两个计数信号量和一个互斥锁：

```text
empty:
    表示还有多少空位
    初始值 = 队列容量 N

full:
    表示已有多少数据
    初始值 = 0

mutex:
    保护队列本身
```

生产者：

```text
P(empty)      // 等空位
lock(mutex)
放入数据
unlock(mutex)
V(full)       // 通知有数据
```

消费者：

```text
P(full)       // 等数据
lock(mutex)
取出数据
unlock(mutex)
V(empty)      // 通知有空位
```

只用一个 `full` 信号量，只能解决“有没有数据可消费”。如果队列有容量限制，还要用 `empty` 控制空位数量；队列本身的修改仍然需要互斥锁保护。

## 原子操作 atomic

普通 `i++` 不是原子操作，至少包含读取、加一、写回几个步骤。

```text
初始 i = 0

线程 A 读到 i = 0
线程 B 读到 i = 0

线程 A 加 1，写回 1
线程 B 加 1，写回 1

执行了两次 i++，结果却是 1
```

`std::atomic<int>` 的自增提供原子读改写语义，保证并发自增不会丢失更新。

注意：

> atomic 不是保证线程不会切换，而是保证该操作对其他线程表现为不可分割。

atomic 和互斥锁的区别：

| 对比 | atomic | 互斥锁 |
|------|--------|--------|
| 适用范围 | 单个变量的简单原子操作 | 一段临界区或多个共享状态 |
| 常见场景 | 计数器、状态标志、引用计数 | 复合业务逻辑、多个变量一致性 |
| 等待方式 | 通常基于 CPU 原子指令或底层实现 | 拿不到锁通常阻塞或等待 |
| 能否保护复合逻辑 | 不适合直接保护复杂逻辑 | 适合 |

面试表达：

> 普通 `i++` 不是原子操作，它包含读取、加一、写回几个步骤。多个线程同时执行时可能交错，导致丢更新。`std::atomic<int>` 的自增提供原子读改写语义，保证并发自增不会丢失更新。atomic 适合单个变量的简单原子操作；互斥锁适合保护一段临界区，维护多个共享状态之间的一致性。

## CAS 和 ABA 问题

CAS 是 Compare And Swap，比较并交换。

```text
内存里的当前值 == 期望旧值？
    是：更新成新值，成功
    否：说明被别的线程改过，失败，通常重试
```

无锁自增可以理解为：

```text
读取 old = count
计算 new = old + 1
CAS(count, old, new)

成功：更新完成
失败：说明别的线程抢先改了 count，重新读再试
```

ABA 问题是指 CAS 只看“当前值是不是 A”，但不知道中间有没有变过：

```text
线程 1 读到值 A，准备 CAS(A -> C)

线程 2 把 A 改成 B
线程 2 又把 B 改回 A

线程 1 再看，发现还是 A
CAS 成功
```

对于普通整数可能没事，但在无锁栈、指针、内存回收等场景可能危险，因为那个 “A 指针” 指向的对象可能已经被弹出、释放、复用过。

常见解决：

- 给值加版本号或时间戳。
- 使用带标记的指针。
- 配合安全的内存回收策略。

面试表达：

> CAS 是比较并交换，只有当内存中的值等于期望旧值时，才把它更新成新值，否则失败重试。ABA 问题是指一个值从 A 变成 B 又变回 A，CAS 只看到当前仍是 A，就误以为没有变化。常见解决方法是加版本号、时间戳，或者使用带标记的指针。

## 死锁

死锁不是忙等待。忙等待是线程一直占 CPU 尝试；死锁是多个线程互相等待对方持有的资源，导致谁都无法继续推进。

典型例子：

```text
线程 A：持有锁 L1，等待锁 L2
线程 B：持有锁 L2，等待锁 L1
```

## 死锁四个必要条件

| 条件 | 含义 |
|------|------|
| 互斥条件 | 资源一次只能被一个线程/进程占用 |
| 占有并等待 | 已经持有一部分资源，同时还在等待其他资源 |
| 不可剥夺 | 资源不能被强制抢走，只能由持有者主动释放 |
| 循环等待 | 存在环形等待链，例如 A 等 B，B 等 A |

注意：

> “不可中断”不是标准四条件。面试里应说“占有并等待”和“不可剥夺”。

## 如何避免死锁

工程里常用方式是破坏四个必要条件之一。

常见做法：

- 固定加锁顺序：所有线程按同一顺序拿锁，破坏循环等待。
- 减少锁嵌套：降低占有并等待的概率。
- 缩小临界区：减少持锁时间。
- 一次性申请多个资源：避免拿着一个锁再等另一个锁。
- `try_lock` + 超时回退：拿不到就释放已有锁，稍后重试。
- 使用 `std::lock` / `std::scoped_lock`：一次安全地锁多个 mutex。

银行家算法是操作系统理论里的死锁避免算法，可以提，但工程开发里更常讲固定加锁顺序、超时回退、减少嵌套锁。

## 线程池

线程池是预先创建并维护一组工作线程，任务到来后放入任务队列，由空闲线程取任务执行。线程执行完任务后不销毁，而是回到线程池继续等待下一次任务。

它解决的问题：

- 避免频繁创建和销毁线程。
- 控制并发线程数量，避免线程过多导致上下文切换、内存占用和调度开销过高。
- 通过任务队列解耦任务提交方和工作线程。

典型结构：

```text
任务队列
互斥锁
条件变量
若干工作线程
停止标志
```

工作线程流程：

```text
while running:
    从任务队列取任务
    没任务就用条件变量等待
    有任务就执行
```

面试表达：

> 线程池是预先创建并维护一组工作线程，任务到来后放入任务队列，由空闲线程取任务执行。线程执行完任务后不销毁，而是继续等待下一次任务。这样可以避免频繁创建和销毁线程的开销，同时限制并发线程数量，防止请求过多时把系统资源打爆。

## 线程池任务队列为什么要有上限

线程池通常要同时限制两个东西：

```text
线程数上限：
    限制并发执行的任务数量

任务队列上限：
    限制等待执行的任务数量
```

如果任务提交速度长期大于处理速度：

```text
任务队列越来越长
请求延迟越来越高
内存占用越来越大
最终可能 OOM 或服务雪崩
```

队列满时要有策略：

- 阻塞提交方。
- 返回失败 / 拒绝请求。
- 丢弃低优先级任务。
- 降级处理。
- 扩容工作线程或扩容服务实例。

面试表达：

> 线程数限制并发执行量，任务队列上限限制等待任务数量。队列有上限是为了防止任务提交速度长期大于处理速度时任务无限堆积，导致延迟升高、内存耗尽甚至服务崩溃。队列满时需要有阻塞、拒绝、丢弃、降级或扩容等策略。

## 容易踩坑的地方

1. 把互斥锁只说成“多个进程抢锁”，忽略多线程是更常见场景。
2. 没说出互斥锁和自旋锁的核心区别：阻塞睡眠 vs 忙等。
3. 自旋锁不适合临界区长，也不适合普通单核场景。
4. 条件变量不是保护资源本身，保护共享状态靠互斥锁。
5. 条件变量 `wait` 会释放锁并睡眠，醒来后重新加锁。
6. 等待条件要用 `while`，不是 `if`。
7. 把死锁和忙等待混在一起。
8. 死锁四条件容易漏掉“占有并等待”和“循环等待”。
9. 工程避免死锁重点是固定加锁顺序、超时回退、减少嵌套锁，不是只背银行家算法。
10. 把二值信号量直接等同于互斥锁，忽略互斥锁所有权语义。
11. 有界生产者消费者只说一个信号量，漏掉 `empty` 和队列互斥保护。
12. 把任务队列上限说成限制线程并发数；线程数上限才限制并发，队列上限限制等待任务数量。
13. 线程池只说复用线程，漏掉控制并发数、任务队列和拒绝/背压策略。
14. 以为 `atomic` 是防止线程切换；实际它提供的是原子读改写语义。
15. 以为多个 atomic 操作组合起来天然整体原子；复合逻辑通常仍需要锁。
16. 只知道 CAS 比较旧值和新值，但不知道 ABA 问题。

## 我的薄弱点

- 三种锁的基本概念知道，但需要主动补“等待方式”和“适用场景”。
- 自旋锁临界区长会浪费 CPU 已理解；单核场景为什么不适合需要继续复测。
- 条件变量知道“条件不满足释放锁、条件满足后唤醒”，但要补牢 `wait` 原子释放锁并睡眠、醒来重新加锁、`while` 防虚假唤醒。
- 死锁概念和忙等待曾混淆，四个必要条件没有一次说全。
- 死锁避免方式需要从理论算法转到工程实践：固定加锁顺序、减少嵌套、`try_lock` 回退。
- 信号量知道是资源访问限制，但要补牢计数器、P/V 操作和所有权区别。
- 生产者消费者模型里容易只想到 `full` 信号量，需要区分 `full` 数据数量、`empty` 空位数量和 `mutex` 队列互斥。
- 线程池主干掌握较好，已经能说出复用线程、减少创建销毁开销；还要补上队列上限、内存风险和拒绝/降级策略。
- `i++` 会拆成读、加、写这点已经理解，但要修正“atomic 不会切换线程”的表达；atomic 保证的是原子语义，不是阻止调度。
- atomic 和锁的边界基本能判断：atomic 管单点原子操作，锁管复合临界区；需要继续练具体场景。
- CAS / ABA 目前属于新知识点，需要后续复测。
