# TCP 粘包与应用层协议

## 一句话理解

TCP 只保证可靠、有序的字节流，不保留应用层消息边界。一次 `send` 和一次 `read` 没有一一对应关系，所以应用层要自己设计协议边界。

常见做法：

```text
4 字节长度字段 + body
```

## 为什么会粘包和半包

- 粘包：多条应用层消息被一次 `read` 读到。
- 半包：一条应用层消息分多次 `read` 才能读完整。
- 本质：TCP 面向字节流，内核只负责按字节连续传输，不知道应用层一条消息从哪里开始、到哪里结束。

## 常见解决方式

| 方式 | 特点 |
|------|------|
| 固定长度 | 简单，但浪费空间，扩展性差 |
| 分隔符 | 文本协议常见，但要处理转义和异常输入 |
| 长度字段 | 最常用，适合二进制协议和面试表达 |

面试优先说长度字段：

> 包头保存 body 长度，接收端为每个连接维护接收缓冲区，从缓冲区里循环解析完整消息。

## 4 字节长度字段解析流程

```text
每个连接维护 recv_buffer

read 到数据后追加到 recv_buffer 尾部

while true:
    如果 recv_buffer 不足 4 字节:
        break，等下次 read

    读取前 4 字节，得到 body_len
    校验 body_len 是否超过最大包长

    如果 recv_buffer 不足 4 + body_len:
        break，等下次 read

    取出完整 body
    从 recv_buffer 移除已消费数据
    交给业务层处理
```

关键点：

- 4 字节头部本身也可能半包，不能假设一次 `read` 一定能读到完整头部。
- 半包不是业务层“解析完再拼接”，而是先放在 `recv_buffer` 里，等数据凑够后再解析。
- 粘包靠 `while` 循环连续解析；半包靠 `break` 留在缓冲区里等下次数据。
- 长度字段必须校验最大值，避免异常包把内存打爆。

## 发送半包怎么处理

`send/write` 也可能只发送一部分数据。返回值 `n > 0` 表示前 `n` 字节已经交给内核，后续只能从第 `n` 字节之后继续发。

正确做法：

```text
要发送的数据放入 output_buffer

调用 send:
    如果返回 n > 0:
        从 output_buffer 移除已发送的 n 字节

    如果 output_buffer 还有剩余:
        监听 EPOLLOUT，等 socket 可写再继续发

    如果全部发完:
        取消 EPOLLOUT，避免一直触发写事件

    如果返回 EAGAIN:
        说明内核发送缓冲区暂时满了，等下次可写
```

读半包靠 `recv_buffer`，写半包靠 `output_buffer`。

## 为什么不能一直 while send

不能在当前线程里一直循环 `send` 到完成，核心原因不是“必然重复发包”，而是：

- 阻塞 socket 可能卡住当前线程。
- 非阻塞 socket 在发不动时会返回 `EAGAIN`，继续循环只会空转耗 CPU。
- Reactor/EventLoop 被卡住后，其他连接的读写事件也处理不了。

如果每次没有根据 `send` 返回值推进偏移量，才会导致重复发送同一段数据。

## output_buffer 越积越多说明什么

说明应用层生产数据的速度大于 socket 实际发送出去的速度，也就是出现了背压。

常见原因：

- 对端读取慢或处理慢。
- 网络拥塞。
- 服务端给这个连接生成响应太快。
- 客户端异常，几乎不读数据。

服务端不能让 `output_buffer` 无限增长，常见处理：

- 设置每个连接的输出缓冲区上限。
- 超过阈值后暂停读事件，降低上游生产速度。
- 丢弃低优先级消息。
- 直接关闭慢连接。

面试表达：

> `output_buffer` 持续增长说明出现背压，发送端生产速度超过了对端或网络的消费能力。服务端要限制每个连接的输出缓冲区大小，必要时暂停读、限流、丢弃低优先级数据或关闭慢连接，防止内存被拖垮。

## 和 HTTP/3 / QUIC 的关系

不要把这个说成“HTTP4”。正式口径里常见的是 HTTP/3：它把 HTTP 语义映射到 QUIC 上，QUIC 基于 UDP，但提供可靠传输、多路复用和流控能力。

面试里可以这样说：

> 自定义 TCP 协议通常靠长度字段解决消息边界；HTTP/3 则是基于 QUIC 的另一套传输模型，不是简单把 TCP 和 UDP 拼在一起。

## 容易踩坑的地方

1. 以为一次 `send` 一定对应一次 `read`。
2. 忽略“4 字节头部本身也可能半包”。
3. 把接收半包和发送半包混在一起。
4. `send` 没发完时，没有根据返回值推进偏移量。
5. 非阻塞 `send` 返回 `EAGAIN` 后继续死循环。
6. `output_buffer` 不设上限，慢连接可能拖垮服务端内存。

## 我的薄弱点

- 已经知道 TCP 是字节流，也知道可以用长度字段定界，但一开始没有把“每个连接维护接收缓冲区 + 循环解析完整帧”说完整。
- 容易把接收半包和发送半包混在一起：接收侧是 `recv_buffer`，发送侧是 `output_buffer`。
- 对 `send` 只发送一部分的核心风险理解偏了：不是重点担心重复发包，而是要处理阻塞、`EAGAIN`、事件循环被卡住和背压。
- 背压概念已经开始建立，但后续还要继续复测“输出缓冲区上限、暂停读、限流、关闭慢连接”这些处理策略。
